## Business Context: Critical Discovery

The Lodgra financial dashboard has **fundamental data source problems**, not just presentation issues.

**What we discovered:**
- Dashboard shows Commission €50 (2% of May revenue)
- Reality: Comissão de Gestão is **10-30% variable per property contract** (management fee percentage)
- €50 is hardcoded or from wrong data source — **NOT reflecting actual management contracts**
- Additionally: 6 data integrity bugs (duplicated metrics, impossible currency conversions, RevPAR math errors)

**Why this is P0:**
- ❌ Dashboard shows FAKE commission numbers to customers
- ❌ Proprietários see €50 commission but actually pay 10-30% per contract
- ❌ Trust destroyed if customer compares dashboard to invoice
- ❌ Demo credibility zero if STR operator sees duplicated metrics (68.1% vs 66%)
- ✅ BUT: STR terminology is correct (ADR, RevPAR, EBITDA, Taxa de Ocupação) — product differentiation exists

**Target Impact:**
- ✅ Dashboard shows REAL commission data from property contracts (not hardcoded €50)
- ✅ All data integrity issues fixed (no impossible values)
- ✅ STR operators see complete, accurate picture (ADR + RevPAR + LOS + correct fees)
- ✅ Customer trust restored before next demo

## Story Map

### 🚨 Phase 0: CRITICAL BLOCKER — Data Source Corruption

#### Story 33.0: Fix Commission Data Source (Dashboard showing fake values)
- **Issue:** Dashboard displays Commission €50 (hardcoded or from wrong source) but should show actual management_fee_percentage from property contract (10-30% variable)
- **Root Cause:** Commission calculation using hardcoded value OR wrong database field, NOT pulling from `user_properties.management_fee_percentage` or contract management
- **Acceptance Criteria:**
  - [ ] Identify where €50 value originates (codebase search + database query review)
  - [ ] Verify `user_properties` table has `management_fee_percentage` field (or create if missing)
  - [ ] Change dashboard to pull actual fee % from property contract
  - [ ] Test with 4+ properties with different fee structures (10%, 15%, 25%, 30%)
  - [ ] Verify calculation: Comissão = Receita × management_fee_percentage (not hardcoded €50)
  - [ ] EUR and BRL calculations both correct
- **Assigned to:** @data-engineer (audit) → @dev (fix logic) → @qa (regression test)
- **Estimate:** 4-5 days
- **Blocks:** Stories 33.2, 33.7 (all commission-related work)
- **Notes:** This is the root cause of dashboard unreliability. All other fixes are secondary to this.

### Phase 1: Data Integrity Fixes (Critical)

#### Story 33.1: Fix Duplicated Taxa de Ocupação Metric
- **Issue:** 68,1% displayed in KPIs block, 66% in commissions block — same metric, different values
- **Root Cause:** TBD (query logic? caching? rounding? different RLS filters?)
- **Acceptance Criteria:**
  - [ ] Single source of truth for Taxa de Ocupação across all dashboard views
  - [ ] Test with known dataset to verify calculation
  - [ ] RLS policies correctly filtered (same user sees same % everywhere)
  - [ ] Verify calculation matches actual booked_nights / available_nights
- **Assigned to:** @data-engineer (diagnosis) → @dev (fix)
- **Estimate:** 3 days
- **Blocked by:** Story 33.0

#### Story 33.2: Fix Currency Conversion Inconsistency
- **Issue:** R$17.230 appears identically in Receita, Lucro, and Repasse — mathematically impossible
- **Root Cause:** Conversion logic using wrong field (e.g., all three using Receita value instead of calculating each)
- **Acceptance Criteria:**
  - [ ] Lucro = Receita - Despesas (BRL conversion correct)
  - [ ] Repasse = Receita - Comissão (BRL conversion correct, uses REAL fee from 33.0)
  - [ ] All three values mathematically distinct in BRL output
  - [ ] Test with EUR/BRL conversion rates
  - [ ] Verify: Receita - Comissão = Repasse (always true)
- **Assigned to:** @data-engineer (query validation) → @dev (conversion logic)
- **Estimate:** 4 days
- **Blocked by:** Story 33.0

#### Story 33.3: Document RevPAR Denominator & Fix Misalignment
- **Issue:** RevPAR €11,88 doesn't match ADR €50,92 × Occupancy 68% (should be ~€34,62)
- **Root Cause:** RevPAR using different denominator (e.g., calendar days vs. available days vs. booked days)
- **Acceptance Criteria:**
  - [ ] RevPAR calculation documented (which denominator? why?)
  - [ ] Formula clearly explained in dashboard tooltip: "RevPAR = ADR × (Booked Days / Available Days)"
  - [ ] If bug: RevPAR value corrected to match formula
  - [ ] If intentional: Clear labels explain non-standard methodology
  - [ ] Test: RevPAR should = ADR × Occupancy (within rounding)
- **Assigned to:** @data-engineer (analysis) → @ux-design-expert (documentation)
- **Estimate:** 2 days
- **Blocked by:** Story 33.0
