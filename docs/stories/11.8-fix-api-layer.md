# Story 11.8: Compliance — Backend & API Layer Remediation

**Epic:** 11 (Compliance & Data Privacy)
**Responsibility:** QA Remediation (Critical & High)
**Status:** Done

---

## Business Context

A análise profunda às rotas de backend do Epic 11 detectou vetores de ataque graves, incluindo Remote Code Execution (RCE) via Formula Injection nos exportadores de CSV, falhas na eliminação completa de dados PII (RGPD Art. 17), vulnerabilidades a timing attacks na autenticação dos Cron jobs e potenciais race conditions. É urgente retificar estes fluxos antes que originem brechas na produção.

---

## Acceptance Criteria

- [x] **AC1 (C2):** Exportações `/api/admin/compliance/csv` implementam sanitização ativa de caracteres sensíveis (`=`, `+`, `-`, `@`) para prevenir vulnerabilidade OWASP Formula Injection.
- [x] **AC2 (C3):** A anonimização do Cron de deletions deve também encriptar ou destruir (nullificar/burlar) totalmente os campos `phone` e `email` do guest side, e o campo de `ip_address` dos `consent_records` passados para tornar o user inteiramente irrastreável após Account Deletion.
- [x] **AC3 (C4):** Endpoint `/api/user/data-export` filtra os `audit_logs` utilizando explicitamente `.eq('organization_id')` para isolar leaks de segurança cross-orgs.
- [x] **AC4 (H2):** O loop de eliminação associado ao Cron Job deve rodar com garantias de atomicidade robustas ou via Database RPC para evitar partial deletions em caso de quebra entre DB drop e o hit da auth admin.
- [x] **AC5 (H3):** Os endpoints `/api/user/delete-request` e `/api/user/cancel-deletion` contêm protecção de Request Rate Limiting para estagnar potenciais DoS attacks em pedidos on/off da framework.
- [x] **AC6 (H4):** O sistema de validação bearer do token cron foi refatorado e migrado para operar através de `crypto.timingSafeEqual()` prevenindo Auth Bypass.
