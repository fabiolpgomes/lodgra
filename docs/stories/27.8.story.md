## 🔍 Problem Statement

The Google Vacation Rentals Feed API (`GET /api/feeds/google-vacation-rentals`) was successfully generating XML feeds with:
- Property names, descriptions
- Pricing, fees, minimum nights
- Geolocation, address, check-in/out times
- Amenities, availability/blocked dates
- Reviews (ratings, counts)

**But it was missing the images.** Without photos, Google couldn't display properties in Hotel Finder widget, severely limiting discovery potential.

### Root Cause Analysis

**Query was pointing to wrong table:**

```typescript
// BEFORE: Looking in property_media (empty table)
supabase.from('property_media')
  .select('url, alt')
  .eq('property_id', prop.id)
  .limit(5)
```

**Actual images stored in:** `property_images` table (25-70 images per property)

**Result:** `images` array was always empty → no `<link rel="image">` tags generated

---

## ✅ Solution Implemented

### 1. **Fixed Query** (Line 109-115)

Changed from `property_media` → `property_images`:

```typescript
const [{ data: images }, ...] = await Promise.all([
  supabase
    .from('property_images')  // ← Fixed: was property_media
    .select('storage_path, alt_text')  // ← Correct columns
    .eq('property_id', prop.id)
    .order('display_order', { ascending: true })  // ← Order by display
    .limit(5),
  // ... rest of queries
])
```

### 2. **Convert Storage Paths to Public URLs** (New function)

Added `convertPropertyImagesToMedia()` function:

```typescript
function convertPropertyImagesToMedia(
  images: Array<{ storage_path?: string; alt_text?: string | null }>
): PropertyMedia[] {
  const supabaseUrl = process.env.NEXT_PUBLIC_SUPABASE_URL
  const storageBucket = 'property-images'

  return images
    .filter((img) => img.storage_path)
    .map((img) => {
      const url = `${supabaseUrl}/storage/v1/object/public/${storageBucket}/${img.storage_path}`
      return {
        url,
        alt: img.alt_text || undefined,
      }
    })
}
```

### 3. **Apply Conversion in Property Enrichment** (Line 152)

```typescript
images: images 
  ? convertPropertyImagesToMedia(images)  // ← Convert storage_path → URL
  : getImagesFromPropertyPhotos(prop.photos),  // ← Fallback for legacy
```

## 📝 Technical Notes

### Why `property_images` and not `property_media`?

- **`property_images`:** Production table with 25-70 images per property, used in booking site
- **`property_media`:** Legacy table, empty in production

The feed generator was originally coded to query `property_media`, which was intended but never populated.

### Image URL Format

Images are stored in Supabase with:
- **Bucket:** `property-images`
- **Path:** `{organization-id}/{property-id}/{image-id}.jpg`
- **Public URL:** `{supabase-url}/storage/v1/object/public/{bucket}/{path}`

Example:
```
https://brjumbfpvijrkhrherpt.supabase.co/storage/v1/object/public/property-images/00000000-0000-0000-0000-000000000001/d574a4e9-87a1-4a31-9561-36936990c193/6bc34ce5-dea4-4c5b-8ee7-ff4197ad070d.jpg
```

### Fallback Logic

If `property_images` returns nothing, the function falls back to legacy `photos` JSON field:

```typescript
images: images ? convertPropertyImagesToMedia(images) : getImagesFromPropertyPhotos(prop.photos)
```

This ensures compatibility with any migration scenarios.
