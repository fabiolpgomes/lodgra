## User Story

As a **Host**, I want **to monitor competitor prices, receive alerts on significant changes, and manage monitoring preferences** so that **I can stay competitive and optimize pricing based on market data**.

---

## Description

Build the frontend UI + background job layer for competitor price monitoring (Story 36.10 backend complete). Implements daily price scraping, alert notifications, competitor dashboard, benchmark reports, and preference management. Mobile-first responsive design with real-time alerts.

**Why:** Backend APIs and market analysis algorithms are ready; hosts need automated monitoring and intuitive UI to act on market data.

### Background Job: Price Scraper

- [ ] **File**: `src/jobs/competitorPriceScraper.ts`
- [ ] **Purpose**: Daily price scraping for all tracked competitors
- [ ] **Trigger**: Scheduled via node-schedule (e.g., runs at 2:00 AM daily)
- [ ] **Logic**:
  1. Fetch all competitors for all active properties
  2. For each competitor URL:
     - Attempt to scrape current price
     - Handle errors gracefully (404, timeout, blocked by site)
     - Extract price value and validate format
  3. Store price snapshot in database (competitor_price_history table)
  4. Calculate price change from previous day
  5. Trigger alert if change ≥ threshold
  6. Update last_scraped timestamp
- [ ] **Error handling**:
  - If scrape fails: Log error, retry up to 3 times with exponential backoff
  - If URL invalid: Mark competitor as "needs attention", notify user
  - If success rate <80%: Send alert to admins
- [ ] **Performance**: Stagger requests (avoid hammering competitor sites)
  - Queue requests with 2-second intervals
  - Set user-agent header (identify as bot: "Mozilla/5.0")
  - Respect robots.txt (if applicable)
- [ ] **Logging**: Log all scraping attempts, successes, failures, errors
- [ ] **Database updates**: Atomic transaction to ensure data consistency

### Background Job: Alert Generator

- [ ] **File**: `src/jobs/alertGenerator.ts` (optional if logic is in scraper)
- [ ] **Purpose**: Generate alerts when significant price changes detected
- [ ] **Logic**:
  1. Compare current price to previous price
  2. Calculate % change
  3. If change ≥ threshold: Create alert record
  4. Deduplicate: Don't alert twice for same competitor on same day
  5. Trigger notification (in-app toast, database record, optional email)
- [ ] **Alert fields**:
  - competitor_id, property_id, old_price, new_price, price_change_percent, timestamp
- [ ] **Email integration** (optional):
  - Queue email job if user has email alerts enabled
  - Aggregate alerts: Send daily/weekly summary instead of real-time

### Data Flow

```
Daily Cron Job (2:00 AM)
  ├─ CompetitorPriceScraper.run()
  │   ├─ Fetch all competitors for all properties
  │   ├─ For each competitor URL:
  │   │   ├─ Scrape price (airbnb/booking/vrbo scraper)
  │   │   ├─ Store price_history record
  │   │   └─ Trigger AlertGenerator if price change ≥ threshold
  │   ├─ Update competitor.last_scraped timestamp
  │   └─ Return: { success_count, failure_count, alerts_created }
  │
  ├─ AlertGenerator.run()
  │   ├─ Fetch new price_history records
  │   ├─ Compare to previous price
  │   ├─ Create alert if % change ≥ threshold
  │   ├─ Deduplicate alerts (same competitor, same day)
  │   ├─ Queue email notification (if user opted in)
  │   └─ Return: { alerts_created }

CompetitorDashboard (User-Facing)
  ├─ useCompetitorMonitoring hook (fetch data on mount)
  ├─ Render:
  │   ├─ CompetitorList (all competitors)
  │   ├─ MarketPositionCard (analysis)
  │   ├─ AlertNotifications (recent alerts)
  │   ├─ Export buttons (CSV, PDF)
  │   └─ Action buttons (Add, Preferences)
  ├─ AddCompetitorModal (on "Add" tap)
  ├─ CompetitorPreferencesModal (on "Preferences" tap)
  ├─ CompetitorDetailView (on row tap)
  └─ Error boundaries + loading states
```

### Scraping Strategy

- **Libraries**: Cheerio (lightweight DOM parser) or Puppeteer (if JavaScript rendering needed)
- **User-Agent**: Set to Mozilla/5.0 to identify as browser
- **Rate Limiting**: Queue requests with 2–3 second intervals to avoid blocking
- **Error Handling**: Retry up to 3 times with exponential backoff (1s, 2s, 4s)
- **Caching**: Store successful prices, use cached value if scrape fails
- **Logging**: Log all attempts, successes, failures for debugging

### Platforms Supported

| Platform | URL Pattern | Scraper | Difficulty |
|----------|------------|---------|-----------|
| Airbnb | airbnb.com/rooms/{id} | JavaScript DOM parsing | Medium |
| Booking.com | booking.com/hotel/{id} | HTML parsing | Medium |
| VRBO | vrbo.com/property/{id} | HTML parsing | Medium |

### Database Considerations

- **competitors table**: Indexed on (property_id, last_scraped) for query efficiency
- **competitor_price_history**: Indexed on (competitor_id, scrape_date) for trend queries
- **competitor_price_alerts**: Indexed on (property_id, created_at) for alert listing
- **Cleanup job**: Delete price_history records >90 days old (weekly)

## Background Job Scheduling

import schedule from 'node-schedule';
import { CompetitorPriceScraper } from './competitorPriceScraper';
import { AlertGenerator } from './alertGenerator';

export function initSchedules() {
  // Run price scraper daily at 2:00 AM
  schedule.scheduleJob('0 2 * * *', async () => {

  });

  schedule.scheduleJob('15 2 * * *', async () => {
    await AlertGenerator.run();
  });

  // Cleanup old data weekly (Sunday at 3:00 AM)
  schedule.scheduleJob('0 3 * * 0', async () => {

  });

### Cron Endpoint (for Vercel)

Vercel doesn't support scheduled background jobs natively, so use:
- Option 1: External cron service (EasyCron, NCronJob) calling `/api/cron`
- Option 2: Vercel Cron (if available in your plan)
- Option 3: AWS Lambda with CloudWatch Events
- Option 4: Firebase Cloud Functions

```typescript
// src/app/api/cron.ts
export async function GET(request: NextRequest) {
  const authHeader = request.headers.get('authorization');
  if (authHeader !== `Bearer ${process.env.CRON_SECRET}`) {
    return new Response('Unauthorized', { status: 401 });
  }

  const jobType = request.nextUrl.searchParams.get('job');
  
  if (jobType === 'scrape') {
    const result = await CompetitorPriceScraper.run();
    return Response.json(result);
  }
  
  if (jobType === 'alerts') {
    const result = await AlertGenerator.run();
    return Response.json(result);
  }

  return Response.json({ error: 'Invalid job type' }, { status: 400 });
}
```

## Error Handling Strategy

### Scraping Errors

| Error | Handling | User Message |
|-------|----------|--------------|
| URL invalid | Skip competitor, mark for review | "Competitor URL needs attention" |
| Network timeout | Retry up to 3 times with backoff | None (use cached price) |
| 404 (page not found) | Mark competitor as invalid | "Competitor URL not found" |
| Price extraction failed | Log error, use cached price | None (use last known price) |
| Rate limited (429) | Increase interval between requests | None (graceful degradation) |

### Alert Handling

| Scenario | Action |
|----------|--------|
| Alert creation fails | Retry next job cycle, log error |
| Email send fails | Queue for retry, show in-app notification |
| High failure rate (>20%) | Send alert to admins |

### User Notifications

- **In-app toast**: For immediate feedback (add, delete, export actions)
- **Alert notifications**: For price changes (persistent in database)
- **Email notifications**: For daily/weekly summaries (optional)

## Risks

| Risk | Mitigation |
|------|-----------|
| Web scraping blocked (403 Forbidden) | Use cached data, implement proxy rotation, graceful degradation |
| Rate limiting (429 Too Many Requests) | Increase intervals between requests, implement backoff strategy |
| Incorrect price extraction | Test scrapers thoroughly, manual price entry option, price validation |
| Competitor URLs become invalid | Validate URLs regularly, mark invalid URLs for user review |
| High traffic on scraper job | Stagger scraping across time, queue-based architecture, load distribution |
| Privacy/Legal concerns | Document that data is public-facing, include disclaimer in app |
| Data inconsistency (race conditions) | Use database transactions, lock-based synchronization |
| Performance: Large number of competitors | Pagination, lazy loading, database indexes, query optimization |

- **Mitigation**: Implement retry logic, logging, monitoring, alerting to admins

- **Challenge**: 1000+ competitors with daily prices = large database

## Notes for Developer

1. **Scraping strategy**: Start with Cheerio (lightweight), upgrade to Puppeteer if needed for JavaScript-heavy sites
2. **Test with sample URLs**: Use real Airbnb/Booking URLs in development for realistic testing
3. **Job logging**: Implement comprehensive logging for debugging (timestamp, competitor_id, success/failure)
4. **Error messages**: Be specific ("Could not extract price from Airbnb URL" vs generic "Error")
5. **Caching strategy**: Store successful prices, retry on cache miss
6. **Rate limiting**: Use queue library (bull, rsmq) to distribute scraping over time
7. **Database cleanup**: Schedule weekly cleanup of price_history records >90 days old
8. **Monitoring**: Alert admins if scraping success rate drops below 80%
9. **Dark mode**: Use CSS variables for chart colors (adapt to theme)
10. **Emoji usage**: Avoid in mobile UI (may not render consistently)
11. **User-Agent rotation**: If blocked frequently, rotate user-agent strings (responsibly)
12. **Privacy notice**: Add disclaimer in app: "We monitor publicly-available competitor pricing data"
