# Story 39.6: Card "Hoje" + Painel de Alertas + Sino de Notificações

**Epic:** 39 (Evolução do Dashboard — Analytics do Gestor)
**Story ID:** 39.6
**Story Points:** 8
**Status:** Ready (PO GO 9/10)
**Priority:** P1 — Grupo E (mesmo agente para os 3 blocos — lógica operacional acoplada)
**Depende de:** [39.1](39.1-schema-foundation-metrics.md) (Done). **Dependência leve de [39.5](39.5-indicador-status-sincronizacao.md)** para o gatilho "falha de sync" do sino — ver Dependencies.

---

## User Story

Como **gestor**, quero um resumo do que precisa da minha atenção hoje (check-ins, check-outs, limpezas, mensagens), um painel com riscos de concentração de receita, e um sino que fica vermelho quando algo precisa de ação (hóspede sem nome, sync falhou, ocupação caindo), para que eu não precise vasculhar cada tela pra descobrir o que está errado.

---

## Por que os 3 blocos numa story só

Card "Hoje" e Painel de Alertas são blocos operacionais adjacentes que compartilham a mesma lógica de "o que precisa de atenção agora" (spec-fonte, Grupo E). O Sino usa parte dos mesmos dados (ex.: falha de sync, ocupação baixa) — separar em 3 agentes geraria 3 implementações divergentes do mesmo conceito de "pendência". Ver `docs/stories/epic-39-dashboard-analytics-evolution.md`.

---

## Description

### A. Card "Hoje"

Novo card, posicionado antes de "Próximas Chegadas" (linha ~736 atual). Mostra, para o dia corrente:
- Check-ins de hoje (`reservations.check_in = hoje`, status confirmado)
- Check-outs de hoje (`reservations.check_out = hoje`, status confirmado)
- Limpezas do dia (`cleaning_tasks` com `scheduled_date = hoje`, já tem `reservation_id` — join direto, conforme pré-requisito da spec-fonte seção 1, já satisfeito pelo schema atual)
- Mensagens/pendências do dia — **se não houver uma fonte de dados clara para "mensagens" no código atual (avaliar durante implementação; não foi encontrada nenhuma tabela de mensageria de hóspede no schema investigado), omitir esse item do card em vez de inventar um dado falso, e sinalizar a Fabio que esse dado não existe ainda.**

### B. Painel de Alertas

Substitui ou expande o bloco "Ações Rápidas" atual (linha ~800-843) — decisão de layout do `@dev`/`@ux-design-expert`; a spec não exige remover "Ações Rápidas", só que o Painel de Alertas exista. Absorve:
- **Concentração por propriedade:** se 1 imóvel > 40% da receita bruta do mês (usar `monthly_property_metrics`, mesma fonte da 39.4). Limiar configurável (constante nomeada, não hardcoded solto), independente do limiar de canal (60%, Story 39.3) — são dois alertas diferentes, não confundir.
- **Concentração por canal:** já implementada na Story 39.3 dentro do próprio card de canal — aqui, se a 39.3 já estiver pronta, apenas **referenciar/reaproveitar** o mesmo cálculo em vez de duplicar; se não, implementar aqui e a 39.3 reaproveita.
- **Faturas de proprietários pendentes de pagamento** — mencionado na spec-fonte (seção 4, item 11) mas **não existe tabela/modelo de dados para isso no schema atual** (investigação não encontrou `owner_invoices` nem equivalente). **Não implementar nesta story** — não inventar uma feature sem lastro de dados. Documentar como pendência a perguntar a Fabio antes de qualquer story futura tentar isso.

### C. Sino de Notificações

Hoje é só um `<button>` decorativo (`page.tsx`, linhas ~426-433: ícone `Bell` + `<span>` dourado fixo, sem lógica). Precisa virar funcional:
- Vermelho quando há qualquer uma das pendências abaixo; clique abre uma box (dropdown/popover) com uma linha por pendência, contexto suficiente para agir sem navegar.
- **Gatilhos:**
  1. **Hóspede com nome placeholder** — o valor real usado no código para fallback é `'Hóspede'` (maiúscula, com acento — não `'hóspede'` minúsculo como a spec-fonte escreveu; conferir a string exata usada em `sync-ical/route.ts:286`, `sync/import/route.ts:142`, `cron/email-parser/route.ts:185`). Há um precedente de query para essa checagem em `src/app/api/admin/audit-ical-data/route.ts` (linhas ~92, ~101, `r.guests.first_name !== 'Hóspede'`) — reaproveitar o padrão, não reinventar. `'Reserved'` também é mencionado na spec — checar se algum caminho de criação ainda deixa esse valor passar (o parser de iCal já filtra `'Reserved'` em `bookingParser.ts:192`, então esse gatilho específico pode já estar coberto na origem; validar se sobra algum caso antes de assumir que precisa de checagem extra).
  2. **Falha de sync** — depende da Story 39.5 gravar em `sync_logs` (hoje o fluxo inbound não grava nada lá — ver Dependencies). Se a 39.5 ainda não estiver pronta quando esta story for implementada, implementar os outros 3 gatilhos e deixar este como `TODO` explícito, não como alerta falso-negativo silencioso.
  3. **Pagamentos pendentes** — reservas com `status = 'pending_payment'` (valor já usado em `api/public/bookings/route.ts`) há mais de X tempo — definir o "X" com Fabio antes de hardcode (não está definido na spec-fonte).
  4. **Ocupação baixa** — alguma propriedade com ocupação abaixo de **30%** nos próximos 30 dias (limiar já definido na spec, seção 3 e seção 5 — não é ponto em aberto). Mensagem identifica o **nome do imóvel** (não uma mensagem genérica). Precisa de query prospectiva (noites reservadas nos próximos 30 dias / 30 dias, por propriedade) — dado diferente do histórico de `monthly_property_metrics` (que é passado), então é uma query nova sobre `reservations` com `check_in`/`check_out` futuros.

## Acceptance Criteria

- [ ] Card "Hoje": check-ins, check-outs e limpezas do dia corrente, com link para o detalhe de cada reserva/tarefa.
- [ ] Card "Hoje" não inventa dado de "mensagens" se não houver fonte real — omite o item e documenta a ausência.
- [ ] Painel de Alertas: concentração por propriedade (limiar 40%, configurável, não hardcoded solto) e concentração por canal (reaproveitando 39.3 se pronta).
- [ ] Painel de Alertas NÃO implementa "faturas de proprietários pendentes" — sem dado para isso, fica fora, com nota documentada.
- [ ] Sino fica vermelho quando: hóspede com nome placeholder (`'Hóspede'` real do código, não `'hóspede'` da spec), falha de sync (se 39.5 pronta), pagamento pendente (com prazo a confirmar com Fabio), ou ocupação baixa (<30% nos próximos 30 dias).
- [ ] Clique no sino abre uma box com uma linha por pendência, incluindo contexto (ex.: nome da reserva/hóspede, nome do imóvel na ocupação baixa) — nunca uma mensagem genérica sem identificar o que está errado.
- [ ] Nenhum destes 4 gatilhos vira um card ou bloco separado além do sino (regra explícita da spec-fonte, seção 3, nota sobre item 6) — reserva sem hóspede, falha de sync, pagamento pendente e ocupação baixa usam **só** o sino.
- [ ] Ocupação baixa identifica o imóvel específico pelo nome, não mensagem genérica.
- [ ] Filtro de propriedade no topo afeta o Painel de Alertas e o Card "Hoje" (o sino é global da organização, não filtrado — confirmar esse entendimento com Fabio se parecer errado).
- [ ] Design segue `docs/design-system/design.md`.
