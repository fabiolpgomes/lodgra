### 2. Timezone Handling: UTC for Financial Calculations

**Priority:** HIGH  
**Effort:** 3 SP  
**Description:**  
Financial calculations currently use browser/server local time. For correctness, should use UTC across:
- Check-in/check-out date parsing
- Month boundary calculations (especially for leap year Feb 29)
- API responses

**Specific Issues:**
- `calculateDaysBetween()` uses `new Date()` (browser TZ)
- Month calculations assume local time zones
- Edge case: User in UTC+8 vs UTC-8 could get different month distribution

**Actions:**
- Audit all date operations in `revenue-calculator.ts`
- Convert to UTC-based calculations: `new Date().toISOString()`
- Add timezone tests (different timezones, leap years)
- Document timezone assumptions in API response headers

### 3. Caching Layer: 1-Hour TTL Implementation

**Priority:** MEDIUM  
**Effort:** 3 SP  
**Description:**  
Story Dev Notes specify 1-hour cache TTL for revenue calculations, but endpoints are currently stateless (no caching).

**Implementation:**
- Add Redis cache (or in-memory for dev) to `/api/dashboard/revenue` and `/api/dashboard/profit`
- Cache key format: `revenue:${currency}:${month}` per Dev Notes
- TTL: 3600 seconds (1 hour)
- Invalidation: On new reservation creation/update

**Monitoring:**
- Log cache hits/misses
- Monitor performance gain (expected: 100ms → 10ms on cache hit)
