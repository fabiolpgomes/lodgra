# Story 36.3: Calendar Manager — Daily Price Grid

**Epic:** 36 (Dynamic Pricing + Calendar Redesign)  
**Story ID:** 36.3  
**Status:** Draft  
**Story Points:** 5  
**Priority:** HIGH  

---

## User Story

As a **Host**, I want **to override prices on specific dates** so that **I can charge different prices for peak seasons or special events**.

---

## Description

Build calendar grid showing month view with daily prices. Users tap any day to open modal and override the price for that date. Grid fetches from `property_daily_prices` endpoint (Story 36.1).

**Why:** Hosts need fine-grained control over pricing for individual dates.

---

## Acceptance Criteria

### Calendar Display

- [ ] Show month calendar (7 columns × 4-5 rows)
- [ ] Header: Month name + ◄ Previous | Next ► navigation buttons
- [ ] Each cell displays: Day number + Price (EUR)
- [ ] Prices loaded from API:
  - If `property_daily_prices` exists for date → use custom price
  - Else → use `base_price` (from `property_prices`)
- [ ] Visual indicators:
  - Booked days: Gray background + guest name
  - Weekend days: Subtle color difference
  - Today: Red circle on day number
- [ ] Navigation: Load new month when ◄ or ► tapped

### Day Cell Interaction

- [ ] Tap any day → Open modal/bottom sheet with:
  - Day label (e.g., "Friday, July 21, 2026")
  - Input: "Preço para este dia" (default = base_price)
  - Button: "Salvar"
  - Button: "Remover override" (only if custom price exists)
  - Close button (X)

### Save Logic

- [ ] On "Salvar":
  - If input price = base_price → DELETE from `property_daily_prices`
  - If input price ≠ base_price → INSERT/UPDATE `property_daily_prices`
  - Grid updates immediately (optimistic UI)
  - Show success message

### Month Navigation

- [ ] Load month data via `GET /api/properties/:id/daily-prices?month=2026-07`
- [ ] Cache month data (don't reload if already loaded)
- [ ] Smooth transition between months

### Responsive Design

- [ ] Mobile (≤768px): Full-width grid
- [ ] Web (>768px): Narrower grid (fits in column layout)

## Risks

| Risk | Mitigation |
|------|-----------|
| Performance with 30+ booked days | Lazy-load reservations, use React.memo on cells |
| Timezone issues in date handling | Always use UTC in DB, format on UI |
| Optimistic UI fails (API error) | Revert to previous state on API error |

## Technical Notes

- Grid layout: CSS Grid (7 columns × 5 rows max)
- Cache months in React Query
- Optimistic updates: Update UI before API call
- Booked dates: Fetch from reservations table (join logic)
