## Story

**Como** utilizador da plataforma,
**Quero** poder exportar todos os meus dados pessoais num formato legível por máquina,
**Para que** eu possa exercer o meu direito à portabilidade de dados conforme RGPD/LGPD.

---

## Contexto de Negócio

RGPD (Art. 20) e LGPD (Art. 18-V) obrigam a disponibilizar ao utilizador uma cópia dos seus dados pessoais em formato estruturado (JSON/CSV). Isto inclui:
- Perfil (nome, email, role)
- Propriedades
- Reservas
- Despesas
- Proprietários
- Logs de auditoria
- Registos de consentimento

### Prazo legal
- RGPD: 30 dias para responder a pedido
- LGPD: 15 dias para responder a pedido
- Implementação self-service elimina carga operacional

## Dev Notes

### Estrutura do JSON de export

```json
{
  "export_date": "2026-04-07T12:00:00Z",
  "user": { "id": "...", "email": "...", "role": "...", "created_at": "..." },
  "properties": [...],
  "reservations": [...],
  "expenses": [...],
  "owners": [...],
  "consent_records": [...],
  "audit_logs": [...]
}
```

### Segurança
- Usar `requireRole()` para autenticação
- Apenas os dados do próprio utilizador (user_id filter)
- Dados da org filtrados por organization_id
- Admin pode exportar para qualquer user da sua org

### Code Quality Assessment

Good implementation of RGPD Art.20 / LGPD Art.18 right to data portability. The parallel `Promise.all` data fetch is efficient and correct. Rate limiting via `audit_logs` is pragmatic — avoids an extra table and leverages existing infrastructure.

The `Content-Disposition: attachment` header ensures the browser downloads the file rather than rendering it inline — correct for this use case.

One observation: the `select('*')` on `properties` and `owners` tables will include all columns. This is intentional for data portability (user is entitled to all their data) but worth documenting in dev notes.
