Dois problemas de error handling corrigidos directamente durante a review; um de validação de input na API também corrigido.

### Refactoring Performed

- **File:** `src/components/features/properties/AmenitiesSelector.tsx`
  - **Change:** Adicionado try/catch em `load()` com estado `error` e verificação `res.ok`
  - **Why:** Falhas de rede ou erros de API causavam crash silencioso; `setLoading(false)` nunca era chamado
  - **How:** Estado `error` renderiza mensagem ao utilizador; `finally` garante que loading termina sempre

- **File:** `src/components/features/properties/AmenitiesSelector.tsx`
  - **Change:** Adicionado try/catch em `save()` com verificação `res.ok`; `setSaved(true)` movido para dentro do `try`
  - **Why:** Falha no PUT mostrava "✓ Guardado" mesmo quando o servidor retornava erro
  - **How:** Utilizador vê mensagem de erro real; `finally` garante `setSaving(false)` sempre

- **File:** `src/app/api/properties/[id]/amenities/route.ts`
  - **Change:** Adicionada validação de input no PUT — verifica que body é array de strings antes de operar na DB
  - **Why:** Input inválido (null, objecto, array com não-strings) causava delete de todas as comodidades seguido de erro DB exposto
  - **How:** Retorna 400 imediatamente se payload inválido

### Performance Considerations

`import * as LucideIcons` importa toda a biblioteca lucide-react no bundle do cliente. É o trade-off necessário para ícones data-driven (nomes vêm da BD). Next.js não consegue fazer tree-shaking de imports dinâmicos por string key. Aceitável para este contexto — não é um hot path e lucide-react já é dependência do projecto.
