## Story

**Quero** gerar tokens de acesso para cleaners sem password,  
**Para que** eles possam aceder ao portal via link seguro.

## Acceptance Criteria (Granular)

### AC1: API endpoint POST /api/cleaning/access-tokens
- [ ] **GIVEN** authenticated manager (admin/manager role)
- **WHEN** POST `/api/cleaning/access-tokens` com body `{ cleaner_id, expires_in_hours }`
- **THEN** response 201 com token (plain text, único request)
- [ ] **AND** token é salvo hashed em DB (SHA-256)

### AC2: Token hash nunca plain text em DB
- [ ] **GIVEN** token gerado
- **WHEN** armazenado em `cleaner_access_tokens.token_hash`
- **THEN** é hash SHA-256, não plain text
- [ ] **AND** impossível reverter hash para token original

### AC3: Verificação de token GET /api/auth/verify-token
- [ ] **GIVEN** token válido
- **WHEN** GET `/api/auth/verify-token?token=xyz123`
- **THEN** response 200 com `{ valid: true, cleaner_id, organization_id }`
- [ ] **AND** token inválido retorna 401 `{ valid: false }`

### AC4: Redirect automático para dashboard
- [ ] **GIVEN** URL `/[locale]/cleaning?token=xyz123`
- **WHEN** carrega página
- **THEN** verifica token, cria session, redirects para `/[locale]/cleaning/dashboard`
- [ ] **AND** token inválido mostra erro: "Token expirado ou inválido"

### AC5: Token expira após X horas
- [ ] **GIVEN** token com `expires_in_hours=24`
- **WHEN** cria token
- **THEN** `expires_at = now() + 24 hours`
- [ ] **AND** após expiração, verify-token retorna 401

### AC6: Logout invalida token imediatamente
- [ ] **GIVEN** cleaner logged in com token
- **WHEN** clica logout
- **THEN** token é marcado como `is_used=true` + `used_at=now()`
- [ ] **AND** token não pode ser reutilizado

### AC7: Rate limiting na verificação
- [ ] **GIVEN** 5 tentativas falhadas de token em 15 min (mesmo IP)
- **WHEN** 6ª tentativa
- **THEN** bloqueia requests desse IP por 15 min
- [ ] **AND** retorna 429 `Too Many Requests`

### AC8: Auditoria de tokens
- [ ] **GIVEN** token criado
- **WHEN** é usado
- **THEN** armazena: ip_address, user_agent (para auditoria)
- [ ] **AND** log em Sentry se pattern suspeito (múltiplos IPs)

### AC9: Token é único + não-reutilizável
- [ ] **GIVEN** mesmo manager gera 2 tokens para mesmo cleaner
- **WHEN** ambos tokens criados
- **THEN** ambos são válidos (não há "único por cleaner")
- [ ] **BUT** uma vez que token é "usado" (login), fica marcado is_used=true

### AC10: Configuração de TTL por organização
- [ ] **GIVEN** org_settings.token_expires_hours = 48
- **WHEN** manager gera token sem especificar expires_in_hours
- **THEN** usa default da org (48h)
- [ ] **AND** manager pode override: `expires_in_hours=12` força 12h

## Integration Verification (IV)

### IV1: Usa Supabase Auth existente
- [ ] **WHEN** token endpoint é chamado
- **THEN** usa `supabase.auth.getUser()` (não novo provider)
- [ ] **AND** verifica role (admin/manager) via RLS

### IV2: RLS policies permitem acesso apropriado
- [ ] **GIVEN** cleaner com token
- **WHEN** acessa `/api/cleaning/dashboard`
- **THEN** RLS permite ler apenas suas tasks (`WHERE cleaner_id = current_user_id`)

### IV3: Não quebra autenticação existente
- [ ] **WHEN** owner/manager login normal (password)
- **THEN** sem mudanças — sistema paralelo de token
- [ ] **AND** ambos tipos de auth coexistem

- **Rate limiting:** Upstash prevents brute force (5/15min per IP)
- **Audit logging:** IP + user agent for suspicious pattern detection
- **Token lifecycle:** Mark used to prevent reuse
- **Organization isolation:** All queries filtered by organization_id

### Decisions Made (YOLO Mode)

1. **Token Format:** 128-char hex (64 bytes) for security over UUID4
2. **Hash Algorithm:** SHA-256 for industry standard + Node.js native support
3. **Rate Limiting:** Upstash (existing project integration) for 5 attempts/15min
4. **Storage:** Plain token never stored — only hash in DB
5. **Expiration:** Per-organization configurable default (24h) with per-token override
6. **Token Extraction:** Query > Header > Cookie priority for flexibility
7. **Middleware:** Reusable wrapper for protecting cleaner endpoints

### Known Limitations

- Upstash Redis configuration required in `.env` for rate limiting
- Rate limiting disabled if Redis connection unavailable (logs warning)
- Token verification scans all tokens (O(n)) — could optimize with indexed hash lookup for very large volumes
- IP detection relies on headers (x-forwarded-for) in production

### Integration Points

- **Supabase Auth:** Uses `supabase.auth.getUser()` for manager verification
- **Organization RLS:** All queries respect organization_id isolation
- **Upstash:** Rate limiting for distributed deployment
- **Database:** Reads from `cleaner_access_tokens` table (created in 29.1)
