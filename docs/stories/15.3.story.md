# Story 15.3: Pull Sync & Painel de Configuração de Canal

## Status
Draft

## Story

**As a** gestor de propriedades,  
**I want** configurar as credenciais do Booking.com no dashboard e sincronizar reservas históricas com um clique,  
**so that** todas as reservas anteriores sejam importadas automaticamente com dados completos.

## Acceptance Criteria

1. Página de configuração de canais acessível em `/[locale]/settings/channels` (ou tab em Settings existente)
2. Utilizador introduz `property_id` externo e `api_key` do Booking.com por propriedade
3. Sistema valida credenciais contra API do Booking.com antes de guardar — exibe erro claro se inválidas
4. Credenciais guardadas de forma segura (nunca expostas no frontend — guardadas como env ou em tabela encriptada)
5. Pull inicial importa reservas dos últimos 90 dias ao activar o canal
6. Indicadores visuais: "Último sync: há X minutos", "X reservas importadas"
7. Botão "Sincronizar agora" dispara pull manual (feedback de loading + resultado)
8. iCal sync existente mantém-se activo para propriedades sem canal API configurado (sem regressão)
9. Feature protegida por variável de ambiente `BOOKING_CHANNEL_ENABLED=true` (off por default)

## Tasks / Subtasks

- [ ] Task 1: Feature flag e guarda de credenciais (AC: 4, 9)
  - [ ] Verificar `process.env.BOOKING_CHANNEL_ENABLED === 'true'` antes de mostrar UI
  - [ ] Criar tabela (via migração) `channel_credentials`: `id`, `organization_id`, `channel_id`, `property_id` (externo), `api_key_encrypted` (text), `created_at`, `updated_at`
  - [ ] OU alternativamente: guardar `api_key` em Vercel env vars por org (decisão: consultar @architect)
  - [ ] Nunca retornar `api_key` em qualquer endpoint GET

- [ ] Task 2: API Route — validar e guardar credenciais (AC: 2, 3, 4)
  - [ ] Criar `POST /api/channels/booking/connect`
  - [ ] Recebe: `{ property_listing_id, external_property_id, api_key }`
  - [ ] Valida credenciais: chamada de teste à API Booking.com (ex: GET /properties/{id})
  - [ ] Se válidas: upsert em `channel_credentials` + upsert em `channel_listings`
  - [ ] Retorna erro 422 com mensagem clara se credenciais inválidas

- [ ] Task 3: API Route — pull sync (AC: 5, 6, 7)
  - [ ] Criar `POST /api/channels/booking/sync`
  - [ ] Recebe: `{ property_listing_id }` (ou sync all para a org)
  - [ ] Usa credenciais guardadas para chamar API Booking.com: GET /reservations?from=90daysago
  - [ ] Para cada reserva: upsert em `reservations` + upsert em `guests` (mesma lógica do webhook)
  - [ ] Actualiza `channel_listings.last_synced_at` e contagem
  - [ ] Retorna `{ synced_count, last_sync_at }`

- [ ] Task 4: UI — Página de configuração (AC: 1, 2, 6, 7)
  - [ ] Criar página `/src/app/[locale]/settings/channels/page.tsx`
  - [ ] OU adicionar tab "Canais" em settings existente (verificar estrutura actual)
  - [ ] Formulário por propriedade: campo `External Property ID` + campo `API Key` (password input)
  - [ ] Botão "Ligar canal" → chama `POST /api/channels/booking/connect`
  - [ ] Estado ligado: mostrar "✅ Booking.com ligado — último sync: há X min — X reservas importadas"
  - [ ] Botão "Sincronizar agora" → chama `POST /api/channels/booking/sync` com loading state
  - [ ] Feedback de erro claro se credenciais inválidas

- [ ] Task 5: Verificar regressão iCal (AC: 8)
  - [ ] Confirmar que `/api/cron/sync-ical` continua a correr para propriedades sem canal API
  - [ ] Confirmar que propriedades com canal API também podem continuar com iCal (não exclusivo)

- [ ] Task 6: Configurar variáveis de ambiente (AC: 9)
  - [ ] Adicionar `BOOKING_CHANNEL_ENABLED=false` ao `.env.local.example`
  - [ ] Documentar: ativar com `BOOKING_CHANNEL_ENABLED=true` após aprovação do Connectivity Program
