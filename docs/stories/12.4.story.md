## Riscos & Mitigação

**Load Test Environment Cost & Performance** — Executar 100 concurrent payments em ambiente staging pode custar dinheiro (Stripe charges por request mesmo em test mode para algumas operações). Load test pode falhar se CI/CD environment não tem recursos suficientes. **Mitigation:** Use `k6` ou `Apache JMeter` simulado (mock Stripe responses) para load testing inicial; load test real com Stripe test mode num horário off-peak; monitorar custo Stripe durante load test.

**Sentry Quota & Alert Fatigue** — Se payment errors spam Sentry excessivamente, quota pode exceder; muitos alertas podem causar alert fatigue (ignorar verdadeiras issues). **Mitigation:** Implementar sampling em Sentry (capture 50% of errors em produção); criar alerting rules específicas (só alerta para CRITICAL ou recurring issues); monitorar Sentry quota usage mensal.

**Test Flakiness & Timing Issues** — E2E tests com Puppeteer podem ser flaky (timing issues, network delays, DOM not ready). Testes passam 90% das vezes, falham aleatoriamente. **Mitigation:** Adicionar retries para E2E tests (TestCafé ou Playwright retry built-in); mock network requests quando possível; usar `waitForNavigation()` explícito antes de assertions.

### Webhook Retry Logic

```typescript
// lib/stripe/webhooks/retry.ts
const MAX_RETRIES = 3;
const BACKOFF_MS = 1000; // 1s, 2s, 4s

async function processWithRetry(fn, retryCount = 0) {
  try {
    return await fn();
  } catch (error) {
    if (retryCount < MAX_RETRIES) {
      await sleep(BACKOFF_MS * Math.pow(2, retryCount));
      return processWithRetry(fn, retryCount + 1);
    }
    throw error;
  }
}
```

## Testing Strategy

**Unit Tests (30+):**
- Signature verification: válida assinatura correcta, rejeita inválida
- Rate limiting: 11th request rejeitado com 429 Too Many Requests
- Retry logic: falha 1ª vez, sucesso na 2ª retry
- Edge cases: network timeout, invalid card, expired trial, duplicate booking

**Integration Tests (20+, Stripe Test Mode):**
- Full subscription flow: signup → create subscription → webhook updates
- Full booking flow: create payment intent → charge.succeeded → payout created
- Webhook retry: send event 3x, all 3 processed correctamente
- Proration calculation: upgrade mid-cycle calcula correctamente

**E2E Tests (5+, Puppeteer):**
- Subscription happy path: signup → choose plan → pay → see confirmation
- Booking happy path: browse property → select dates → checkout → see confirmation
- Upgrade path: upgrade from Starter to Professional → see price change
- Refund flow: guest initiates refund → see refund status

**Load Testing (k6):**
- Simulate 100 concurrent booking payments (5 min duration)
- Verify 0 failures, <1s p99 latency
- Monitor Sentry/logs for errors
- Verify all 100 payments recorded in database

**Manual Testing:**
- Stripe test mode: verify all webhook signatures valid
- PCI audit: grep logs para card numbers, CVV, PII (deve estar vazio)
- Sentry testing: trigger intentional error, verify captured in dashboard
- Rate limit testing: send 11 requests in 1 minute, verify 11th rejected

**⚠️ Nota:** **Stripe test mode EXCLUSIVAMENTE** até aprovação final. Produção sandbox: usar real Stripe account em restricted mode (read-only) antes de full production launch.
