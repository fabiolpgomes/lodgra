## Story

**Como** administrador do sistema,
**Quero** que usuários deletados possam ser re-registrados com o mesmo email e que novos usuários recebam senha provisória,
**Para que** eu tenha controle total sobre gerenciamento de usuários e possa impor mudança de senha no primeiro login.

---

## Contexto de Negócio

### Problema 1: Usuários Deletados Ainda Existem em Supabase Auth
- Quando um usuário é deletado, apenas o registro em `user_profiles` é removido
- O usuário continua existindo em `supabase.auth.users` (Supabase Auth)
- Tentar criar uma nova conta com o mesmo email resulta em erro: **"A user with this email address has already been registered"**
- Impossível reutilizar emails de usuários deletados

**Impacto:**
- Usuários deletados ocupam emails permanentemente
- Admin não consegue criar novo usuário com email anterior
- Workflow de gerenciamento de usuários quebrado

### Problema 2: Falta Senha Provisória
- Novo usuário criado via admin não recebe senha provisória
- Sem email de boas-vindas com senha temporária
- Usuário não consegue fazer login na primeira vez
- Sem fluxo de "força trocar senha no primeiro login"

**Impacto:**
- Usuários novos não conseguem acessar o sistema
- Falta de segurança: senhas padrão nunca definidas
- Experiência de usuário ruim

---

## Critérios de Aceitação

- [x] **AC1** — Ao deletar usuário, remover também da Supabase Auth (via Admin API ou RPC)
- [x] **AC2** — Email do usuário deletado fica disponível para re-registro
- [x] **AC3** — Ao criar novo usuário via admin, gerar senha provisória automática (ex: `TempPass2026!A1b2`)
- [x] **AC4** — Enviar email com senha provisória ao novo usuário
- [x] **AC5** — Ao fazer login com senha provisória, forçar mudança de senha antes de acessar dashboard
- [x] **AC6** — Campo `password_reset_required` ou similar em `user_profiles` para marcar se precisa trocar senha
- [x] **AC7** — Middleware ou rota de change-password como primeiro step após login

## Dev Notes

### Root Cause Analysis

**Problema 1 - Usuários Deletados:**
- Arquivo: `src/app/api/user/[id]/delete/route.ts` (se existir) ou mutation em Supabase
- Supabase Auth é independente da RLS policy — deletar `user_profiles` não deleta o usuário de `auth.users`
- Solução: Usar Supabase Admin API (`adminAuthClient.admin.deleteUser()`) ou RPC function que deleta do auth

**Problema 2 - Senha Provisória:**
- Arquivo: `src/app/admin/users/new/page.tsx` (form de criar usuário)
- API route: `src/app/api/admin/users/route.ts` (POST para criar)
- Precisa: gerar password aleatória → criar user no auth → enviar email com password → marcar password_reset_required

### Files to Modify

1. **User deletion endpoint** (find and update)
   - DELETE `/api/admin/users/[id]` or equivalent
   - Call Supabase Admin API to delete from auth

2. **User creation endpoint** (create or update)
   - POST `/api/admin/users` or `/api/organization/members/invite`
   - Generate provisional password
   - Create user in Supabase Auth
   - Send email with password
   - Set `password_reset_required: true`

3. **User schema migration** (if needed)
   - Add `password_reset_required BOOLEAN DEFAULT false` to `user_profiles`

4. **Login/Auth flow** (update)
   - After successful login, check `password_reset_required`
   - Redirect to `/auth/change-password` if true
   - Only allow access after password changed

5. **Change password page** (create or update)
   - Form to change password from provisional to permanent
   - Must be accessible to newly created users

6. **Email template** (create)
   - New user welcome email with provisional password
