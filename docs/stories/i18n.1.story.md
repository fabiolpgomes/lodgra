## Story

**Como** equipa de produto,  
**Quero** configurar a infraestrutura i18n com next-intl para suportar múltiplas línguas e locales,  
**Para que** as stories seguintes (i18n.2, i18n.3, i18n.4) tenham uma base sólida e escalável.

---

## Contexto de Negócio

O plano estratégico da Home Stay inclui expansão para 3 mercados:
- 🇵🇹 **Portugal** (PT) — existente
- 🇧🇷 **Brasil** (PT-BR) — crescimento
- 🇺🇸 **USA** (EN-US) — enterprise

Atualmente, o projeto tem uma implementação manual e limitada (`src/lib/i18n/messages.ts` com apenas 2 chaves). Precisamos de uma solução profissional, escalável e integrada com Next.js.

**Alvo:** Usar **next-intl** (framework oficial Next.js para i18n) com estrutura de ficheiros modular, suporte a 4+ locales, routing automático.

## Dev Notes

### Dependências
- `next-intl` v3+ (latest)

### Referência
- [next-intl Docs](https://next-intl-docs.vercel.app/)
- Strategic Roadmap: `docs/STRATEGIC_ROADMAP.md` — Section "LANGUAGES & LOCALIZATION"

### Decisões de Arquitectura
1. **Ficheiros JSON** vs código — JSON é mais fácil para traduções futuras (Crowdin, etc)
2. **Server Components** — next-intl é nativo em Next.js 13+
3. **Default locale** — `pt` (Portugal) como fallback
4. **URL routing** — `/[locale]/path` mais SEO-friendly que domínios diferentes

### Compatibilidade
- Next.js 15 (app router)
- TypeScript strict mode
- Supabase auth (sem conflito)
- Existing middleware (será refatorado depois)

### Implementation Notes

1. **Middleware Integration Strategy**: next-intl middleware is called first in the middleware chain, ensuring locale is extracted from the URL before any other middleware logic runs.

2. **Layout Restructure**: All pages are now under [locale] segment, enabling per-locale layouts and metadata generation.

3. **Backwards Compatibility**: Old routes without locale prefix are automatically redirected by next-intl middleware to /[locale]/path.

4. **Message Structure**: 5 namespaces for translations (common, dashboard, navigation, forms, errors) allow for modular organization and can be extended in i18n.2.

5. **Type Safety**: TypeScript types ensure correct translation key usage and enable IDE autocomplete for translation keys.

6. **Server vs Client**: 
   - Server components use getTranslations() helper from next-intl/server
   - Client components use useTranslations() hook from next-intl/client
