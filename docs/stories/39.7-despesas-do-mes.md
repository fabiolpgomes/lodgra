## User Story

Como **gestor**, quero ver as despesas por propriedade lançadas no mês, espelhando o card "Receita do Mês", para que eu tenha visibilidade de custo sem precisar entrar no módulo de Despesas.

---

## Description

Novo card "Despesas do Mês", espelhando visualmente o card "Receita do Mês" existente (`page.tsx`, linhas ~542-583) — mesmo padrão de layout, breakdown por moeda.

**Reaproveitamento importante:** a query de despesas do mês corrente **já existe** em `page.tsx`, linhas 308-322 (`currentMonthExpenses` / `monthExpensesByCurrency`, da tabela `expenses` — despesas por propriedade, já vinculada a `property_id` + `date`, pré-requisito da spec-fonte seção 1 já satisfeito antes desta epic). Essa query hoje alimenta o cálculo de "Lucro Real", mas a Story 39.2 troca a fonte do Lucro Real para `company_expenses` (despesas da empresa) — **depois da 39.2, essa query fica sem uso na página**. Esta story reaproveita exatamente essa query (ou uma variação dela, se a 39.2 a tiver removido) para alimentar o novo card, em vez de escrever uma nova do zero.

## Dev Notes

**Estado encontrado ao iniciar:** a Story 39.2 já estava mesclada em `main` e já havia removido completamente a antiga query/variável `monthExpensesByCurrency` de `expenses` que alimentava o Lucro Real — confirmado via `grep -n "expenses" page.tsx` antes de codar, exatamente como o brief instruía. A query de `expenses` foi então escrita do zero (não é um bloqueio, como já previsto no brief e no Technical Notes da story).

**Query nova (`expenses`, despesas por propriedade):**
```ts
const { data: monthExpensesRows, error: monthExpensesError } = propertyIds.length > 0
  ? await supabase
      .from('expenses')
      .select('amount, currency, property_id, expense_date')
      .eq('organization_id', organizationId)
      .in('property_id', propertyIds)
      .gte('expense_date', currentMonthStart.toISOString().slice(0, 10))
      .lte('expense_date', currentMonthEnd.toISOString().slice(0, 10))
  : { data: null, error: null }
```
- Filtra por `organization_id` (coluna existente na tabela `expenses` desde a migration `20260325020000_add_organization_id_to_tables.sql`, indexada) **e** por `property_id in propertyIds` — o segundo filtro é o que garante que o filtro de propriedade do topo da página afeta o card (AC).
- Agrupamento por moeda usa a mesma prioridade já estabelecida no arquivo para outras seções (`propertyCurrencyMap[property_id]` → `expense.currency` → `orgCurrency`), reaproveitando o helper `propertyCurrencyMap` já calculado no topo do arquivo (linha ~124) em vez de duplicar lógica.
- Variáveis nomeadas deliberadamente `monthPropertyExpensesByCurrency` / `propertyExpensesEntries` (prefixo "property") para não confundir com `company_expenses`/`companyExpensesCurrentMonth` usadas no card "Lucro Real" (AC explícito contra essa confusão).

**Posicionamento do card:** decisão de design não coberta explicitamente pela story — o card foi colocado no mesmo grid de "Receita do Mês" e "Lucro Real" (em vez de em bloco separado abaixo), por ser o agrupamento mais coerente visualmente e por instrução do orquestrador de manter as edições próximas a essa área. Isso exigiu mudar o grid container de `lg:grid-cols-2` para `lg:grid-cols-3`. Validado que o build de produção compila sem erros e que os três cards mantêm o mesmo padrão visual (ícone circular, badge de moeda, tipografia) em qualquer largura ≥1024px (breakpoint `lg`), inclusive ≥1280px (AC da epic). Nenhuma badge MoM/YoY foi adicionada ao card, conforme Scope OUT da story.

**Não executado nesta sessão:** verificação manual do valor do card contra soma manual de `expenses` numa organização real/staging (DoD item 5) — requer acesso a dados reais que não estavam disponíveis nesta sessão autônoma. Recomendado para @qa ou verificação manual do Fabio antes do deploy.

**Pronta para QA:** sim, com a ressalva acima documentada.
