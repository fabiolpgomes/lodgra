# Story 13.1 — Observabilidade: Sentry + Structured Logging

## Objetivo
Implementar error tracking e logging estruturado para ter visibilidade total de erros em produção, permitindo diagnóstico rápido de problemas antes que os utilizadores reportem.

## Acceptance Criteria

- [x] Sentry SDK instalado e configurado (Next.js integration: client + server + edge)
- [x] `instrumentation.ts` configurado para Sentry server-side (Next.js 15 pattern)
- [x] `sentry.client.config.ts` e `sentry.server.config.ts` configurados
- [x] `sentry.edge.config.ts` para middleware/edge functions
- [x] Error boundary global com reportagem automática ao Sentry
- [x] Source maps enviados ao Sentry no build (via `@sentry/nextjs` plugin)
- [x] Structured logging (JSON) nos API routes com `requestId` para correlação
- [ ] Variáveis de ambiente: `SENTRY_DSN`, `SENTRY_AUTH_TOKEN`, `SENTRY_ORG`, `SENTRY_PROJECT`
- [x] Breadcrumbs automáticos para navegação e API calls
- [x] Filtro de PII: não enviar tokens, passwords ou dados pessoais ao Sentry
- [x] Build PASS, Lint PASS
- [ ] Verificar que erros em dev aparecem na dashboard Sentry

## Notas Técnicas

- Usar `@sentry/nextjs` (package oficial com suporte App Router)
- `instrumentation.ts` é o entry point para server-side no Next.js 15
- Sampling rate: 1.0 em dev, 0.2 em produção (ajustar conforme volume)
- Performance monitoring: traces para API routes e page loads
- Não enviar erros de rate limiting (429) como bugs — são esperados

## Dependências
- Conta Sentry (free tier: 5K errors/mês)
- Variáveis de ambiente no Vercel
