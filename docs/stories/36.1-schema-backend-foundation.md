# Story 36.1: Schema + Backend Foundation

**Epic:** 36 (Dynamic Pricing + Calendar Redesign)  
**Story ID:** 36.1  
**Status:** Ready for Review  
**Story Points:** 8  
**Priority:** HIGH  

---

## User Story

As a **Host**, I want **to store and manage pricing configuration** so that **I can set base prices, weekend prices, discounts, and availability rules for my properties**.

---

## Description

Implement the complete database schema and API endpoints needed for pricing configuration management. This is the foundation for all pricing features (daily overrides, discounts, availability). All pricing data flows through these endpoints.

**Why:** Without schema + API, no pricing configuration can be stored or retrieved.

---

## Acceptance Criteria

### Database Schema

- [ ] Create table `property_prices`:
  - `id` (UUID PK)
  - `property_id` (UUID FK, NOT NULL)
  - `base_price` (DECIMAL(10,2), NOT NULL)
  - `weekend_price` (DECIMAL(10,2), optional)
  - `created_at`, `updated_at`
  - Unique constraint on `property_id`

- [ ] Create table `property_discounts`:
  - `id` (UUID PK)
  - `property_id` (UUID FK, NOT NULL)
  - `discount_type` (ENUM: weekly, monthly, excellent_guest, last_minute, advance)
  - `percentage` (INT CHECK 0-100)
  - `min_nights` (INT, for duration-based discounts)
  - `conditions` (JSONB, for complex rules)
  - `created_at`, `updated_at`

- [ ] Create table `property_availability`:
  - `id` (UUID PK)
  - `property_id` (UUID FK, NOT NULL, UNIQUE)
  - `min_nights` (INT default 1)
  - `max_nights` (INT default 365)
  - `advance_notice_days` (INT default 0)
  - `notice_for_same_day` (TIME default '00:00')
  - `preparation_days` (INT default 0)
  - `created_at`, `updated_at`

- [ ] Create table `property_daily_prices`:
  - `id` (UUID PK)
  - `property_id` (UUID FK, NOT NULL)
  - `date` (DATE NOT NULL)
  - `price` (DECIMAL(10,2))
  - `created_at`, `updated_at`
  - Unique constraint on `(property_id, date)`

### API Endpoints

- [ ] `GET /api/properties/:id/prices` → Return { base_price, weekend_price }
- [ ] `POST /api/properties/:id/prices` → Create/update prices
- [ ] `GET /api/properties/:id/discounts` → Return all discount rules
- [ ] `POST /api/properties/:id/discounts` → Create discount rule
- [ ] `PUT /api/properties/:id/discounts/:discountId` → Update discount
- [ ] `DELETE /api/properties/:id/discounts/:discountId` → Remove discount
- [ ] `GET /api/properties/:id/availability` → Return availability settings
- [ ] `PUT /api/properties/:id/availability` → Update availability
- [ ] `GET /api/properties/:id/daily-prices?month=2026-07` → Return daily overrides for month
- [ ] `POST /api/properties/:id/daily-prices` → Set daily price override
- [ ] `DELETE /api/properties/:id/daily-prices/:date` → Remove daily override

### Response Format

All endpoints return:
```json
{
  "success": true,
  "data": { /* response */ },
  "error": null
}
```

Errors return appropriate codes: 400 (bad request), 404 (not found), 422 (validation error)

### Validation

- [ ] All endpoints validate property ownership (user must own property)
- [ ] Price inputs reject negative values
- [ ] Discount percentage 0-100 range enforced
- [ ] Availability min_nights ≤ max_nights validation
- [ ] All database operations include error handling

|------|-----------|

| API security (property ownership) | Always check `user_id` matches property owner |
| Performance with 1000+ discount rules | Index on `(property_id, discount_type)` |

## Technical Notes

- Foreign keys should have ON DELETE CASCADE for cleanup
- Index on `property_id` for fast lookups (queries)
- Cache discount rules in memory (optional, for performance)
- API responses should include pagination for daily-prices if >100 records
