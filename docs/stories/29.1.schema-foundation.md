## Contexto de Negócio

O portal de cleaners requer novos tipos de dados: tarefas de limpeza, templates de checklist, itens de checklist, fotos de prova, e tokens de acesso para cleaners. Todos os dados devem ser multi-tenant (`organization_id`) e ter RLS configurado.

### O que já existe
- Tabela `user_profiles` com campo `guest_type` (values: 'staff', 'owner')
- Supabase Storage configurado (usado em despesas e propriedades)
- RLS padrão por `organization_id` em todas as tabelas principais

### O que falta
- Novo `guest_type` value: `'cleaner'`
- Tabela `cleaning_tasks` — tarefa de limpeza por reserva/propriedade
- Tabela `cleaning_checklist_templates` — templates criados pelo gestor
- Tabela `cleaning_checklist_items` — itens do template
- Tabela `cleaning_checklist_responses` — respostas do cleaner por tarefa
- Tabela `cleaning_photos` — fotos de prova por tarefa
- Tabela `cleaner_access_tokens` — tokens de autenticação sem password
- Storage bucket `cleaning-photos`

## Acceptance Criteria (Granular)

### AC1: Migração criada com schema completo
- [ ] **GIVEN** migração em `supabase/migrations/`
- **WHEN** `supabase db push` é executado
- **THEN** todas as 6 tabelas + storage bucket são criadas em staging

### AC2: `cleaning_tasks` tabela com campos corretos
- [ ] **GIVEN** schema `cleaning_tasks`
- **WHEN** query `SELECT * FROM cleaning_tasks WHERE organization_id = 'xxx'`
- **THEN** retorna: id, organization_id, property_id, reservation_id, cleaner_id, checklist_template_id, status, scheduled_date, scheduled_time, notes, completed_at, created_at, updated_at

### AC3: `cleaning_checklist_templates` tabela
- [ ] **GIVEN** schema `cleaning_checklist_templates`
- **WHEN** manager cria template
- **THEN** armazena: id, organization_id, property_id (nullable), name, description, is_active, created_at

### AC4: `cleaning_checklist_items` tabela
- [ ] **GIVEN** schema `cleaning_checklist_items`
- **WHEN** items são adicionados a template
- **THEN** armazena: id, template_id (FK), label, category, is_required, order_index

### AC5: `cleaning_checklist_responses` tabela
- [ ] **GIVEN** schema `cleaning_checklist_responses`
- **WHEN** cleaner marca item como completo
- **THEN** armazena: id, task_id (FK), item_id (FK), is_done, done_at, cleaner_notes

### AC6: `cleaning_photos` tabela
- [ ] **GIVEN** schema `cleaning_photos`
- **WHEN** foto é uploaded
- **THEN** armazena: id, task_id (FK), file_path (storage URL), uploaded_at, uploader_id (FK user_profiles)

### AC7: `cleaner_access_tokens` tabela
- [ ] **GIVEN** schema `cleaner_access_tokens`
- **WHEN** token é gerado
- **THEN** armazena: id, cleaner_id (FK), token_hash (sha256), created_at, expires_at, is_used, used_at, ip_address (auditoria)

### AC8: RLS policies em todas as tabelas
- [ ] **GIVEN** RLS enabled em todas as 6 tabelas
- **WHEN** cleaner tenta acessar task de outra organização
- **THEN** query falha com "Violation of row-level security policy"

### AC9: Storage bucket `cleaning-photos` criado com RLS
- [ ] **GIVEN** bucket `cleaning-photos` com RLS
- **WHEN** cleaner faz upload
- **THEN** ficheiro armazenado em path: `cleaning-photos/{organization_id}/{task_id}/{filename}`
- [ ] **AND** RLS policy permite apenas leitura/escrita por cleaner próprio + manager da org

### AC10: Índices criados para performance
- [ ] **GIVEN** índices em:
  - `cleaning_tasks(organization_id, property_id)`
  - `cleaning_tasks(organization_id, cleaner_id, scheduled_date)`
  - `cleaning_checklist_items(template_id)`
  - `cleaning_photos(task_id, uploaded_at)`
  - `cleaner_access_tokens(token_hash)`
- **WHEN** queries executam
- **THEN** tempo < 100ms (verificado via EXPLAIN ANALYZE)

### AC11: Migração testada sem erros
- [ ] **GIVEN** migração em staging
- **WHEN** `supabase db push` em staging
- **THEN** sucesso sem warnings
- [ ] **AND** rollback também funciona: `supabase db reset`

### AC12: Foreign keys com referential integrity
- [ ] **GIVEN** foreign keys definidos:
  - cleaning_tasks.organization_id → organizations.id
  - cleaning_tasks.property_id → properties.id
  - cleaning_tasks.reservation_id → reservations.id (nullable)
  - cleaning_tasks.cleaner_id → user_profiles.id (nullable)
  - cleaning_checklist_items.template_id → cleaning_checklist_templates.id (ON DELETE CASCADE)
  - cleaning_checklist_responses.task_id → cleaning_tasks.id (ON DELETE CASCADE)
  - cleaning_checklist_responses.item_id → cleaning_checklist_items.id (ON DELETE CASCADE)
  - cleaning_photos.task_id → cleaning_tasks.id (ON DELETE CASCADE)
  - cleaner_access_tokens.cleaner_id → user_profiles.id
- **WHEN** tentamos deletar task
- **THEN** respostas + fotos são deletadas automaticamente (CASCADE)

### AC13: `user_profiles` atualizado com guest_type='cleaner'
- [ ] **GIVEN** coluna `guest_type` em user_profiles
- **WHEN** valor é 'cleaner'
- **THEN** RLS rules reconhecem cleaner como role válido

## Integration Verification (IV)

### IV1: Schema integra com existing tables
- [ ] **WHEN** query joins cleaning_tasks + properties + reservations + user_profiles
- **THEN** foreign keys funcionam sem erros

### IV2: RLS não quebra existing queries
- [ ] **WHEN** manager queries `SELECT * FROM properties`
- **THEN** RLS policies não afectam (já existem)
- [ ] **AND** nova RLS em cleaning_* não interfere

### IV3: Storage bucket isolado
- [ ] **WHEN** cleaner_A faz upload para cleaning-photos
- **THEN** cleaner_B não consegue ler ficheiros de cleaner_A (RLS bucket policy)

### IV4: Migrations são versionadas
- [ ] **GIVEN** ficheiro `supabase/migrations/{timestamp}_create_cleaning_schema.sql`
- **WHEN** future developers correm `supabase db push`
- **THEN** migration é aplicada sequencialmente
