## Story

**Como** proprietário de alojamentos,
**Quero** que o sistema leia automaticamente os emails de confirmação de reserva do Airbnb, Booking.com e Flatio,
**Para que** as reservas sejam criadas no sistema sem intervenção manual, poupando tempo e evitando erros de introdução de dados.

---

## Contexto de Negócio

As plataformas enviam emails de confirmação quando uma reserva é efectuada:

```
Nova reserva confirmada
Hóspede: João Silva
Check-in: 10 Maio
Check-out: 15 Maio
Valor: €850
```

Actualmente o proprietário tem de introduzir estes dados manualmente. Com o parsing automático, o sistema:
1. Conecta à caixa de Gmail via OAuth2
2. Lê emails de remetentes conhecidos (Airbnb, Booking, Flatio)
3. Extrai os dados com Claude Haiku
4. Cria a reserva em estado Draft para revisão
5. Notifica o proprietário via Resend

Arquitectura completa: `docs/architecture/email-parsing.md`

---

## Critérios de Aceitação

- [ ] **AC1** — Proprietário consegue conectar a sua conta Gmail via OAuth2 (botão em Settings), com scope `gmail.readonly`
- [ ] **AC2** — Após conexão, o sistema mostra estado "Conectado (email)" e data da última sincronização
- [ ] **AC3** — Cron job corre a cada 5 minutos e lê emails não lidos de `automated@airbnb.com`, `noreply@booking.com`, `noreply@flatio.com`
- [ ] **AC4** — Claude Haiku extrai correctamente: `guest_name`, `checkin_date`, `checkout_date`, `amount`, `currency`, `platform`, `confirmation_code`
- [ ] **AC5** — Reserva criada com `status=draft` em Supabase, associada à organização correcta
- [ ] **AC6** — Email de notificação enviado via Resend ao proprietário com link para confirmar/rejeitar a reserva
- [ ] **AC7** — Emails já processados não são processados novamente (deduplicação via `message_id`)
- [ ] **AC8** — Proprietário consegue desconectar o Gmail (revoga token no Google + apaga registo em BD)
- [ ] **AC9** — Tokens OAuth2 armazenados encriptados em Supabase (pgcrypto AES-256)
- [ ] **AC10** — Se a extracção falhar (email malformado, campo em falta), o erro é registado em `email_parse_log` e o proprietário é notificado

### Schema de Base de Dados

```sql
-- Ligações Gmail por organização
CREATE TABLE email_connections (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id uuid NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  email           text NOT NULL,
  access_token    text NOT NULL,  -- encriptado pgcrypto
  refresh_token   text NOT NULL,  -- encriptado pgcrypto
  token_expiry    timestamptz NOT NULL,
  scope           text NOT NULL DEFAULT 'gmail.readonly',
  connected_at    timestamptz NOT NULL DEFAULT now(),
  last_sync_at    timestamptz,
  UNIQUE(organization_id)
);

-- Log de emails processados (deduplicação + auditoria)
CREATE TABLE email_parse_log (
  id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id uuid NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  message_id      text NOT NULL,
  received_at     timestamptz NOT NULL,
  platform        text,
  status          text NOT NULL,  -- parsed | skipped | error
  parsed_data     jsonb,
  reservation_id  uuid REFERENCES reservations(id),
  error_message   text,
  created_at      timestamptz NOT NULL DEFAULT now(),
  UNIQUE(message_id, organization_id)
);
```

### Tratamento de Erros

- Token expirado → refresh automático via `refresh_token`
- Refresh falha → notificar proprietário para reconectar
- Claude falha → registar em `email_parse_log.status = 'error'`, notificar proprietário
- Email já processado → `status = 'skipped'` (deduplicação silenciosa)

### Code Quality Findings

#### ✅ STRENGTHS
1. **Architecture:** Clean separation — OAuth2 client, Gmail API wrapper, parsing, persistence
2. **Security:**
   - RLS policies on both tables (SELECT/INSERT/UPDATE/DELETE)
   - Admin/manager role checks on all OAuth routes
   - Token encryption before storage
   - CRON_SECRET validation
3. **Error Handling:** Comprehensive try-catch in cron job, graceful degradation on Google API errors
4. **Deduplication:** UNIQUE constraint + sophisticated retry logic for failed/deleted reservations
5. **Testing Ready:** Verbose console logging for debugging (email.id, body length, parsed data)
6. **Email Extraction:** Handles text/plain, text/html, multipart MIME types correctly
7. **Token Refresh:** Automatic refresh 5 minutes before expiry

#### ⚠️ CONCERNS

1. **Missing AC6 Implementation — Email Notifications**
   - Story requires: "Email de notificação enviado via Resend ao proprietário"
   - Code does not include: Resend integration or notification logic
   - **Impact:** Users won't be notified of auto-parsed reservations
   - **Action:** Add Resend email notification after successful parsing/creation

2. **AC5 Status Mismatch — Draft vs Confirmed**
   - Story AC5 says: "status=draft"
   - Code sets: `status: 'confirmed'`
   - **Question:** Is this intentional auto-confirmation? Verify with product intent.
   - **Options:**
     - Keep confirmed (user sees reservation immediately)
     - Change to draft (user must approve before visibility)

3. **Crypto Key Derivation — Potential Issue**
   - Current: `Buffer.from(KEY_HEX.slice(0, 32), 'utf8')`
   - Risk: If KEY_HEX is hex-encoded string (recommended), this treats it as UTF-8 text
   - **Question:** Is KEY_HEX meant to be:
     - Raw 32-character UTF-8 string? (current implementation)
     - Hex-encoded representation? (should use `.from(KEY_HEX, 'hex')`)
   - **Fix:** Clarify and update if using hex encoding

4. **Parser Error Handling — Silent Null Returns**
   - `parseReservationEmail()` throws on Claude API error but returns null on invalid JSON
   - This causes `email_parse_log.status = 'error'` even though parsing succeeded (just no fields)
   - Minor: Consider distinguishing "parsing failed" from "parsing succeeded but fields missing"

5. **No Rate Limiting on Gmail API**
   - Code fetches up to 50 messages per sync with no rate limiting
   - Gmail API has rate limits — may hit throttling with many orgs
   - Low priority for MVP, but monitor in production
