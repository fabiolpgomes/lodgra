# Story 13.5 — Security Headers & Hardening

## Objetivo
Implementar security headers HTTP e realizar audit de segurança para proteger contra ataques comuns (XSS, clickjacking, MIME sniffing), endurecendo a aplicação para produção.

## Acceptance Criteria

- [x] Security headers via middleware `applySecurityHeaders()`:
  - [x] `Strict-Transport-Security` (HSTS): `max-age=63072000; includeSubDomains; preload`
  - [x] `X-Content-Type-Options`: `nosniff`
  - [x] `X-Frame-Options`: `DENY`
  - [x] `Referrer-Policy`: `strict-origin-when-cross-origin`
  - [x] `Permissions-Policy`: desactivar câmara, microfone, geolocation (não usados)
  - [x] `Content-Security-Policy`: policy adequada ao stack (Supabase, Stripe, Sentry, Vercel)
- [x] `npm audit` executado — zero vulnerabilidades high/critical
- [x] Audit de variáveis `NEXT_PUBLIC_*` — verificar que nenhuma expõe secrets
- [x] Rate limiting review nos endpoints públicos (`/api/stripe/booking-webhook`, `/p/[slug]`)
- [x] Dependency update: packages com vulnerabilidades conhecidas actualizados
- [ ] Headers verificados via securityheaders.com — score A ou A+
- [x] Build PASS, Lint PASS

## Notas Técnicas

- Security headers já existiam em `src/lib/middleware/security-headers.ts` — actualizados
- CSP agora inclui: Sentry (`*.sentry.io`, `*.ingest.sentry.io`), Stripe frames, Supabase WebSocket (`wss://`)
- Nonce passado ao CSP para inline scripts (GA, JSON-LD)
- `npm audit fix` reduziu de 12 vulnerabilidades (6 high) para 4 low (jest-environment-jsdom, dev only)
- NEXT_PUBLIC_* audit: todas safe (Supabase anon key, Stripe publishable key, GA ID, app URL)
- Rate limiting adicionado a: booking webhook (50/min), consent API (15/min), public API (30/min)
