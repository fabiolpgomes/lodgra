# Story 11.7: Compliance — Security & DB Layer Remediation

**Epic:** 11 (Compliance & Data Privacy)
**Responsibility:** QA Remediation (Critical & High)
**Status:** Done

---

## Business Context

Following the QA and architecture review, several critical security vulnerabilities and risks of GDPR/LGPD violations were identified in the Database layer of our compliance features. Specifically, the lack of multi-tenant isolation (`organization_id`) in compliance tables, incorrect RLS policies that allow unauthorized modifications, and inconsistent cascading rules must be fixed to protect PII data and ensure strict multi-tenant isolation.

---

## Acceptance Criteria

- [x] **AC1 (C1):** `consent_records` e `deletion_requests` passam a ter um campo `organization_id` (mandatory, references `organizations(id)`).
- [x] **AC2 (C1):** Políticas RLS nestas duas tabelas garantem estritamente isolamento multi-tenant (admins só acedem aos dados da sua org_id).
- [x] **AC3 (H1):** Política de `UPDATE` na tabela `deletion_requests` tem `WITH CHECK (status = 'cancelled' AND user_id = auth.uid())` para prevenir users de alterarem o próprio pedido para 'completed'.
- [x] **AC4 (H7):** Foreign keys de user nas tabelas `consent_records` e `deletion_requests` usam `ON DELETE SET NULL` para preservar o histórico (audit trail) caso o utilizador base seja removido acidentalmente ou via processo atómico do Auth.
