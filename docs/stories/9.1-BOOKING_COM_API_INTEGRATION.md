# Story 9.1: Booking.com API Native Integration

## Overview
Implement bidirectional Booking.com API integration for real-time reservation sync, price/availability updates, and webhook-based notifications.

## Context
Currently, the system pulls Booking.com reservations via iCal (read-only). To enable:
- Real-time reservation notifications
- Push price & availability updates
- Centralized channel management

We need native Booking.com Extranet API integration.

## Implementation Notes

### Webhook Signature Validation
Booking.com uses HMAC-SHA256 for webhook authentication. The signature is sent in the `X-Booking-Signature` header and is computed as:
```
signature = Base64(HMAC-SHA256(payload, BOOKING_WEBHOOK_SECRET))
```

### Reservation Payload Structure
Expected webhook payload format:
```json
{
  "reservation_id": "string",
  "property_id": "string",
  "guest_name": "string",
  "check_in": "YYYY-MM-DD",
  "check_out": "YYYY-MM-DD",
  "number_of_guests": "number",
  "status": "NEW|MODIFIED|CANCELLED",
  "timestamp": "ISO-8601"
}
```

### Rate Limiting Strategy
- Use Upstash Redis with sliding window algorithm
- Limit: 5 requests per minute per property_id
- Fall back to in-memory store if Redis unavailable
- Return 429 status with Retry-After header on limit exceeded

### Error Handling
All errors must be logged with:
- Request ID for traceability
- Organization ID for multi-tenant isolation
- Property ID for debugging
- Original payload for audit trail

### Testing Strategy
1. Mock Booking.com API responses
2. Test signature validation with valid/invalid signatures
3. Test duplicate detection with repeat reservation IDs
4. Test error scenarios (network failure, malformed payload, rate limit)
5. Test RLS isolation (verify organization_id propagation)
