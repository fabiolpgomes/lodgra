# Story 31.4: Branded Email Confirmations

**Epic:** Epic 31 — Multi-Tenant Infrastructure (Branded Direct Booking)  
**Status:** Ready  
**Priority:** High  
**Type:** Backend + Email Templates  
**Assigned to:** @dev  
**Complexity:** Medium (T-shirt: M)

---

## Story

**As** a Lodgra product owner,  
**I want** companies to receive booking confirmation emails with their own branding (logo, colors, custom message),  
**So that** customers feel they're booking with the company, not through a third-party platform.

---

## Context

Stories 31.2 and 31.3 customize the booking page visually. Story 31.4 extends that to email communications. When a customer books through `pousada.lodgra.io/booking`, they should receive a confirmation email branded as from "Pousada" with Pousada's logo and colors.

**Current State:**
- Booking confirmations sent via email service (likely SendGrid or similar)
- Emails use Lodgra branding
- No company-level customization

**Desired State:**
- Emails dynamically rendered with company branding
- Company logo in email header
- Company colors applied to email buttons/accents
- Company can customize confirmation message
- Custom from-address optional (e.g., noreply@pousada.lodgra.io)

## Dev Notes

### Technical Approach

**1. Database Migration**
```sql
CREATE TABLE organization_email_templates (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID NOT NULL UNIQUE REFERENCES organizations(id) ON DELETE CASCADE,
  confirmation_subject TEXT DEFAULT 'Booking Confirmation',
  confirmation_message TEXT,
  from_email TEXT DEFAULT 'noreply@lodgra.io',
  from_name TEXT,
  reply_to_email TEXT,
  include_company_logo BOOLEAN DEFAULT true,
  footer_text TEXT,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE TABLE email_sent (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID REFERENCES organizations(id),
  booking_id UUID REFERENCES bookings(id),
  customer_email TEXT NOT NULL,
  template_type TEXT DEFAULT 'confirmation',
  sent_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);
```

**2. Email Template Rendering**
- Use mjml library (MJML) for email markup
- Create template files: `src/lib/email/templates/booking-confirmation.mjml`
- Template supports Handlebars syntax for variables
- Render with company branding colors/logo

**3. Backend Logic**
```typescript
// Pseudocode
async function sendBookingConfirmation(booking, organization) {
  const template = await getEmailTemplate(organization.id);
  const branding = await getBranding(organization.id);
  
  const emailHtml = renderMJML('booking-confirmation.mjml', {
    ...booking,
    ...template,
    logo: branding.logo_url,
    primaryColor: branding.primary_color,
  });
  
  await sendEmail({
    from: template.from_email,
    to: booking.customer_email,
    subject: template.confirmation_subject,
    html: emailHtml,
  });
  
  await logEmailSent(organization.id, booking.id, booking.customer_email);
}
```

## Risks & Mitigations

### CRITICAL

**R1: Email Spoofing - from_email Domain Ownership Not Verified**
- **Severity:** CRITICAL
- **Probability:** High
- **Impact:** Phishing attacks, brand hijacking, legal liability
- **Description:** Admin can set `from_email` to "noreply@yourcompetitor.com" without verification. Lodgra would then send emails appearing to come from competitor, enabling phishing or brand impersonation.
- **Mitigation:**
  - Require DKIM/SPF verification for custom from_email domains
  - Admin uploads SPF/DKIM record proof (text file or screenshot) before custom from_email is enabled
  - Default: only allow `noreply@{orgSlug}.lodgra.io` (verified at signup)
  - If custom domain required: Vercel SendGrid integration must validate domain ownership via DNS TXT record
- **Checklist:**
  - [ ] Custom from_email requires domain verification
  - [ ] SPF/DKIM records checked before enabling
  - [ ] Default from_email is organization-owned subdomain
  - [ ] Test spoofing attempt rejected (e.g., "noreply@fake.com" blocked)

**R2: GDPR Compliance - Unsubscribe & PII Handling**
- **Severity:** CRITICAL
- **Probability:** High (if not handled)
- **Impact:** GDPR fines (4-20% of revenue), legal exposure, user rights violation
- **Description:** Booking confirmation emails must include unsubscribe link (GDPR requirement). Customer email and personal data must not be stored longer than necessary. Email audit logs could expose PII.
- **Mitigation:**
  - Every email MUST include unsubscribe link in footer
  - Unsubscribe link: `https://lodgra.io/unsubscribe?token={signed-jwt}`
  - Email_sent logs: do NOT store customer name, phone, payment details (only: customer_email, sent_at, status)
  - Retention policy: Delete email_sent records older than 90 days
  - Admin dashboard hides PII from logs (shows only email_count, delivery_rate)
- **Checklist:**
  - [ ] Unsubscribe link included in every email template
  - [ ] Email_sent table has RLS policy (only org admin can view)
  - [ ] PII fields not stored in email logs
  - [ ] 90-day retention policy enforced
  - [ ] GDPR audit pass: unsubscribe works, PII protected

### HIGH

**R3: Email Delivery Failures - No Retry Logic**
- **Severity:** HIGH
- **Probability:** High
- **Impact:** Lost bookings, customer frustration, business loss
- **Description:** If SendGrid/email service is temporarily down during booking confirmation, email is lost with no retry. Customer never receives confirmation.
- **Mitigation:**
  - Implement exponential backoff retry logic: 3 retries with delays (5s, 30s, 5min)
  - Store email_sent record BEFORE sending (set status='pending')
  - Background job polls pending emails every minute, retries failures
  - Alert Ops if email_sent.status='failed' after 3 retries
- **Checklist:**
  - [ ] Retry logic: 3 attempts with exponential backoff
  - [ ] Email_sent record created before sending (status='pending')
  - [ ] Background job re-attempts failed emails
  - [ ] Failed emails after 3 retries logged and alerted
  - [ ] Load test: email service down, verify retries succeed when service recovers

**R4: Email Rendering Issues Across Clients**
- **Severity:** HIGH
- **Probability:** Medium
- **Impact:** Broken emails, poor UX, lost conversion
- **Description:** MJML templates may render differently in Outlook, Gmail, Apple Mail (different CSS support, mobile rendering, image handling).
- **Mitigation:**
  - Test email rendering in 5 clients: Gmail, Outlook (web), Apple Mail, Yahoo, mobile (iOS)
  - Use MJML best practices: avoid complex CSS, use table-based layout
  - Preload fallback fonts (system fonts)
  - Test with real-world data (long property names, multi-line addresses)
- **Checklist:**
  - [ ] Email rendering tested in Gmail, Outlook, Apple Mail
  - [ ] Mobile rendering tested (Gmail mobile, Apple Mail mobile)
  - [ ] All links clickable, images load in all clients
  - [ ] Text content is readable even without CSS

**R5: Rate Limiting on Email Sends**
- **Severity:** HIGH
- **Probability:** Medium
- **Impact:** Email queue overflow, service degradation
- **Description:** Malicious actor or bug could trigger 1000s of email sends per second, overwhelming SendGrid quota or mailboxes.
- **Mitigation:**
  - Rate limit: 100 emails/min per organization
  - Burst limit: max 10 emails/second (prevents queue overflow)
  - Per-customer limit: max 5 emails/day to same customer (prevents spam)
  - Return 429 Too Many Requests when limit exceeded
  - Alert Ops if org approaches 80% of daily quota
- **Checklist:**
  - [ ] Per-org rate limit enforced: 100/min
  - [ ] Burst limit enforced: 10/sec
  - [ ] Per-customer limit enforced: 5/day
  - [ ] Test: send 150 emails/min → 51st email rejected with 429

### MEDIUM

**R6: Template Variable Injection - Missing Sanitization**
- **Severity:** MEDIUM
- **Probability:** Medium
- **Impact:** HTML injection, rendering breaks, UX degradation
- **Description:** If `{{propertyName}}` contains HTML characters (e.g., "Beach & Resort <luxury>"), could break email layout or inject HTML.
- **Mitigation:**
  - Sanitize all template variables before rendering (HTML escape: `&`, `<`, `>`, quotes)
  - Use Handlebars built-in escaping: `{{variable}}` (always escapes HTML)
  - Triple-brace syntax `{{{variable}}}` only for explicitly safe HTML
  - Test: property name with "<script>" → verify escaped in email
- **Checklist:**
  - [ ] Template variables sanitized (HTML escaped)
  - [ ] Test XSS payload in property name → escaped in email
  - [ ] Triple-brace syntax never used for user-generated content

**R7: Validation of Subject & Message Length**
- **Severity:** MEDIUM
- **Probability:** Low
- **Impact:** Email truncation, poor UX
- **Description:** If subject is 500 chars (AC5 says max 100), email clients truncate subject. If message is 10K chars, renders poorly.
- **Mitigation:**
  - API validates: subject max 100 chars, message max 5000 chars
  - Admin dashboard shows character counter and live preview
  - Return 400 with error if exceeds limit
- **Checklist:**
  - [ ] Subject length validated (max 100 chars)
  - [ ] Message length validated (max 5000 chars)
  - [ ] Admin dashboard shows char counter
  - [ ] Test: 101-char subject rejected with 400

**R8: Performance - Email Template Rendering**
- **Severity:** MEDIUM
- **Probability:** Low
- **Impact:** Slow email sends, delayed delivery
- **Description:** If MJML rendering takes 5+ seconds per email (complex template, slow regex), send 100 emails takes 8+ minutes.
- **Mitigation:**
  - Pre-compile MJML template at startup (not on every email)
  - Render in < 100ms per email (benchmark with MJML CLI)
  - Use async rendering: render in background job, not in booking confirmation request
- **Checklist:**
  - [ ] MJML template pre-compiled at startup
  - [ ] Render time benchmarked: < 100ms per email
  - [ ] Rendering async (background job or queue)

### LOW

**R9: PII in Email Audit Logs**
- **Severity:** LOW
- **Probability:** Low
- **Impact:** Privacy exposure if logs compromised
- **Description:** Email_sent table could be viewed by ops, potentially exposing customer email addresses.
- **Mitigation:**
  - Email_sent RLS policy: only org admin can view own org's logs
  - Ops dashboard redacts email addresses (shows only domain: "*@gmail.com")
  - Logs retained only 90 days
- **Checklist:**
  - [ ] Email_sent table has RLS policy limiting to org members
  - [ ] Ops dashboard redacts email addresses
  - [ ] Logs auto-deleted after 90 days

**R10: Email Tracking & Analytics Implementation**
- **Severity:** LOW
- **Probability:** Low
- **Impact:** Feature limitation, not critical
- **Description:** Admin may want to see email open rates, click tracking (SendGrid supports this). Feature not in AC8 but might be expected by users.
- **Mitigation:**
  - Document in admin dashboard: "Email tracking not currently supported (roadmap)"
  - If tracking needed: SendGrid webhook integration required (future story)
- **Checklist:**
  - [ ] Admin dashboard clarifies: "Email delivery status only (opens/clicks not tracked)"
  - [ ] Roadmap documented for future tracking feature
