## 🎯 Objetivo

Refatorar o sistema de sincronização iCal para usar o **identificador único de reserva da plataforma** (Booking ID, Airbnb Listing ID, etc) como o primary reference, em vez de gerar emails fake e UIDs genéricos.

**Impacto**: Elimina duplicações, re-importações de canceladas, e facilita rastreamento de origem.

---

## 🔴 Problema

### Situação Atual

```
Booking envia iCal:
  UID: 6816972454@booking.com (muda ou não é único)
  DESCRIPTION: (pode ser vazio ou genérico)

Lodgra importa como:
  external_id = "6816972454@booking.com"
  guest_email = "imported-{timestamp}-{random}@lodgra.local" ← PROBLEMA!
  guest_name = "Hóspede Importado" ← GENÉRICO

Quando Booking reencaminha (mesmo evento):
  UID pode ser diferente OR
  Lodgra não encontra match → cria DUPLICATA
  Result: "Hóspede Importado" aparece múltiplas vezes
```

### Cenário de Reprodução

1. Booking envia reserva: `6816972454` (13/08-18/08, Laurence BAREILLE)
2. Lodgra importa como `external_id="6816972454@booking.com"` + `guest_email="imported-...@lodgra.local"`
3. Usuário **deleta** (era falso, valor €0)
4. Booking cancela a reserva
5. Nova sincronização: **reserva volta** (porque iCal ainda tem o evento, mesmo cancelado)
6. Usuário atualiza manualmente: Nome, Valores, Hóspedes
7. Próxima sincronização: **atualiza corretamente** ou **cria outra duplicata**?

---

## ✅ Solução

### Arquitetura Proposta

```typescript
// Antes
{
  external_id: "6816972454@booking.com",
  guest_email: "imported-{timestamp}-{random}@lodgra.local",
  guest_name: "Hóspede Importado"
}

// Depois
{
  external_id: "booking_6816972454",  ← Identificador único e rastreável
  booking_reference: "6816972454",     ← Platform-specific ID
  booking_source: "booking",           ← Plataforma
  guest_email: "imported-...@lodgra.local" ou real do Booking se disponível
  guest_name: "Hóspede Importado" (será preenchido manualmente pelo usuário)
}
```

### Benefícios

| Benefício | Impacto |
|-----------|---------|
| **Identificador único estável** | Sem duplicatas quando UID muda |
| **Rastreamento de origem** | `booking_6816972454` é imediatamente reconhecível |
| **Atualizações sem duplicação** | Match baseado em `booking_reference`, não email fake |
| **Facilita futuro sync bidirecional** | Poder enviar cancelamentos de volta ao Booking |
| **Auditoria melhorada** | Saber exatamente qual ID da plataforma corresponde |

### Subtask 2: Refatorar Sync Logic
**File**: `src/app/api/cron/sync-ical/route.ts`

```typescript
// Antes
const { data: existingReservation } = await supabase
  .from('reservations')
  .select('id')
  .eq('external_id', event.uid)  // ← UID pode mudar!
  .single()

// Depois
const bookingId = extractBookingId(event.description)
const lookupKey = bookingId ? `booking_${bookingId}` : event.uid

const { data: existingReservation } = await supabase
  .from('reservations')
  .select('id')
  .eq('external_id', lookupKey)  // ← Estável!
  .single()
```
