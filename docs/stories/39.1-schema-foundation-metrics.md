# Story 39.1: Fundação de Dados — Schema + monthly_property_metrics

**Epic:** 39 (Evolução do Dashboard — Analytics do Gestor)
**Story ID:** 39.1
**Status:** Done
**Story Points:** 5
**Priority:** P0 — bloqueia 39.2, 39.3, 39.4, 39.6, 39.7

---

## User Story

Como **gestor (Algarve Home Stay)**, quero que as reservas tenham dados completos de canal, desconto e taxa de serviço, e uma agregação mensal por propriedade pronta, para que as métricas de comparação temporal (MoM/YoY), ADR/RevPAR e mix de canais do dashboard sejam confiáveis e não exijam recálculo em tempo real a cada carregamento.

---

## Description

Esta story cobre a Seção 1 ("Pré-requisitos de dados") da spec-fonte (`docs/specs/lodgra-dashboard-spec-consolidada.md`), **já reduzida** ao que realmente falta no schema atual — parte do que a spec pedia já existe:

**Já existe (não recriar):**
- `reservations.commission_amount`, `commission_rate`, `commission_calculated_at` + view `commission_summary` (migration `20260326020000_add_commission_tracking.sql`) → cobre o "comissao_valor" da spec.
- `properties.management_percentage` (migration `20260313010000_add_management_percentage.sql`).
- `properties.cleaning_fee`/`cleaning_fee_type`, `properties.pet_fee`/`pet_fee_type` (migration `20260506_01_property_amenities_rooms_fees.sql`) → valores-base de taxa de serviço.
- `reservations.booking_source` → canal.
- `expenses` (property-level, com `property_id` + `expense_date`) e `company_expenses` (org-level) → ambos já vinculados corretamente.

**Falta implementar (escopo real desta story):**
1. `reservations.discount_amount` — valor de desconto aplicado à reserva.
2. `reservations.service_fee_amount` — soma das taxas de serviço aplicáveis (limpeza + animais, etc.), **copiada da propriedade no momento da criação da reserva**, não recalculada retroativamente se o valor-base da propriedade mudar depois.
3. `monthly_property_metrics` — materialized view (seguir o padrão já usado em `commission_summary`) agregando por `organization_id` + `property_id` + mês: receita bruta, noites vendidas, noites disponíveis, nº de reservas confirmadas, nº de canceladas. Base para os badges MoM/YoY da Story 39.2 e para concentração/mix de canais das Stories 39.3/39.6.
4. Relatório de qualidade de dados: % de reservas com `booking_source` ou `created_at` ausentes — não é migration, é uma verificação a ser documentada no Dev Notes desta story, para orientar as stories seguintes sobre exclusão/sinalização de dados incompletos.

**Por quê:** sem isso, ADR/RevPAR, badges MoM/YoY, mix de canais e concentração (Stories 39.2, 39.3, 39.4, 39.6) não têm base de dados para calcular corretamente, e recalcular tudo em tempo real a cada carregamento do dashboard seria caro e nunca daria valor histórico correto para reservas antigas.

## Acceptance Criteria

### Migration — reservations

- [x] `ALTER TABLE reservations ADD COLUMN IF NOT EXISTS discount_amount NUMERIC(12,2) DEFAULT 0` — nunca `NULL` implícito tratado como erro; default 0 é aceitável aqui (diferente da regra de "nunca assumir zero" que se aplica a receita/canal ausente, não a desconto ausente, que É zero por padrão).
- [x] `ALTER TABLE reservations ADD COLUMN IF NOT EXISTS service_fee_amount NUMERIC(12,2) DEFAULT 0`.
- [x] Identificar o(s) ponto(s) de criação de reserva no código (sync iCal e/ou fluxo manual — localizar via grep, não assumir caminho) e preencher `service_fee_amount` a partir de `properties.cleaning_fee` + `properties.pet_fee` (respeitando `cleaning_fee_type`/`pet_fee_type` — `per_stay` vs `per_night` × noites) no momento da criação da reserva. **6 pontos de criação encontrados e todos modificados** — ver Dev Notes.
- [x] Reservas existentes: backfill de `service_fee_amount` usando o valor **atual** da propriedade é aceitável como aproximação documentada (a spec não exige reconstrução do valor histórico exato para reservas passadas — só exige que, daqui para frente, mudanças no valor-base não afetem reservas já criadas). Implementado como `UPDATE` idempotente dentro da própria migration.

### monthly_property_metrics

- [x] Criar `monthly_property_metrics` como materialized view (mesmo padrão de `commission_summary`), com colunas mínimas: `organization_id`, `property_id`, `property_name`, `metric_month` (primeiro dia do mês), `gross_revenue` (soma de `total_amount`, Receita Bruta, excluindo `status = 'cancelled'`), `nights_sold`, `available_nights`, `booking_count`, `cancelled_count`.
- [x] Índice único em `(organization_id, property_id, metric_month)` para permitir refresh concorrente (`REFRESH MATERIALIZED VIEW CONCURRENTLY`), seguindo o padrão de `idx_commission_summary_unique`.
- [x] Índice de consulta em `(organization_id, metric_month DESC)` para lookups do dashboard.
- [x] Reservas com `booking_source` ou `total_amount` ausentes **não** entram silenciosamente como zero — excluir do agregado e contar separadamente (coluna `incomplete_data_count` ou equivalente), para permitir sinalização nas stories seguintes.

### Qualidade de dados (não-migration, verificação)

- [x] Relatório (pode ser output em texto no Dev Notes desta story, não precisa ser feature de UI) com: % de reservas totais com `booking_source IS NULL`, % com `created_at IS NULL`. Se houver volume relevante de dados incompletos, documentar como isso afeta os cards das próximas stories. **Query preparada, não executada — sem acesso a banco (local ou remoto) neste ambiente; ver Dev Notes.**

### Regras gerais

- [x] Migração é aditiva e reversível — instruções de rollback documentadas no cabeçalho do arquivo SQL, no mesmo formato usado em `20260326020000_add_commission_tracking.sql`.
- [x] Nenhuma duplicação de `commission_amount`/`commission_rate`/`commission_summary` já existentes.
- [x] RLS: `monthly_property_metrics` é materialized view — RLS não se aplica diretamente (mesma ressalva documentada em `commission_summary`); controle de acesso via camada de API/query, filtrando por `organization_id`.

### 3. Decisões técnicas na migration (`monthly_property_metrics`)

- **Atribuição de mês (`metric_month`):** usa `DATE_TRUNC('month', check_in)` — toda a receita e noites da estadia são atribuídas ao mês do check-in, sem dividir proporcionalmente estadias que atravessam a virada do mês. Simplificação deliberada para manter a view como uma agregação direta (mesmo padrão de `commission_summary`, que também não faz split por noite). Se o volume de estadias longas cruzando mês for relevante, isso pode inflar/distorcer levemente o mês de check-in às custas do mês de check-out — a mitigar em 39.2 se os números de ADR/RevPAR parecerem descolados da realidade.
- **`available_nights`:** calculado como dias corridos do mês civil (`EXTRACT(DAY FROM ...)`), **sem** descontar `calendar_blocks` (bloqueios manuais ou sincronizados da plataforma). Ou seja, é uma capacidade "teórica" do mês, não a disponibilidade real líquida de bloqueios. Motivo: `calendar_blocks.block_type` distingue `manual` de `platform_sync`, e não ficou claro sem mais investigação se blocks `platform_sync` duplicam períodos já cobertos por reservas (o que causaria dupla contagem se subtraídos ingenuamente). Documentando aqui para que 39.2 (que consome esta view para Ocupação/RevPAR) decida conscientemente se isso é suficiente ou se precisa de ajuste — não é um "esquecimento", é uma decisão de escopo para não expandir a materialized view além do que a story pediu.
- **Linhas geradas:** a view só produz uma linha por `(organization_id, property_id, metric_month)` que tiver **pelo menos uma reserva** naquele mês (não é um cross-join propriedade × calendário). Meses sem nenhuma reserva não aparecem — consistente com a regra global da epic de "nunca assumir zero para dado ausente": ausência de linha deve ser tratada como "sem dado", não como "0 reservas confirmadas".
- **`incomplete_data_count`:** conta reservas não-canceladas com `booking_source IS NULL OR total_amount IS NULL` — excluídas de `gross_revenue`/`nights_sold`/`booking_count`, contadas à parte, conforme AC.
- **Backfill de `service_fee_amount`:** feito via `UPDATE ... WHERE service_fee_amount = 0` (idempotente — seguro reexecutar), usando o valor **atual** de `properties.cleaning_fee`/`pet_fee`, não o valor histórico no momento da criação (que não existia antes desta migration). Documentado como aproximação aceita pela story.

### 4. Relatório de qualidade de dados — query pronta, NÃO executada

Sem acesso a um banco Supabase local (Docker não estava disponível neste ambiente — `supabase status` falhou por falta do daemon Docker) nem permissão para tocar em banco remoto, a query abaixo não pôde ser executada. Fica pronta para rodar antes do QA gate:

```sql
SELECT
  COUNT(*) AS total_reservations,
  COUNT(*) FILTER (WHERE booking_source IS NULL) AS missing_booking_source,
  ROUND(100.0 * COUNT(*) FILTER (WHERE booking_source IS NULL) / NULLIF(COUNT(*), 0), 2) AS pct_missing_booking_source,
  COUNT(*) FILTER (WHERE created_at IS NULL) AS missing_created_at,
  ROUND(100.0 * COUNT(*) FILTER (WHERE created_at IS NULL) / NULLIF(COUNT(*), 0), 2) AS pct_missing_created_at
FROM reservations;
```

Nota: `created_at` tem `DEFAULT now()` desde a criação da tabela (migration `20260100000000_bootstrap_core_tables.sql`), então a expectativa é 0% ou próximo disso — mas a story pede o número real, não uma estimativa, por isso não foi marcado como "assumido 0%".

## Runbook — Reconciliar histórico de migrations após aplicação manual

**Quando usar:** toda vez que uma migration SQL for aplicada diretamente no SQL Editor do dashboard Supabase (em vez de `supabase db push`), a tabela de bookkeeping `supabase_migrations.schema_migrations` fica desatualizada — o schema real muda, mas o CLI continua achando que a migration está pendente. Isso já aconteceu com pelo menos 41 migrations deste projeto desde 19/04/2026.

**Passo 1 — nunca repare sem confirmar primeiro que o schema real bate com o esperado.** `migration repair` só edita a tabela de bookkeeping — não verifica nada, não roda SQL de schema. Se você marcar como "applied" uma migration que na verdade não rodou por completo, o CLI para de tentar aplicá-la para sempre e o schema fica quebrado silenciosamente. Para confirmar, rode uma query read-only checando os objetos que a migration deveria ter criado, por exemplo (adaptar colunas/tabelas por migration):

```sql
SELECT
  EXISTS (SELECT 1 FROM information_schema.columns WHERE table_name='<tabela>' AND column_name='<coluna>') AS has_coluna,
  EXISTS (SELECT 1 FROM pg_matviews WHERE matviewname='<view>') AS has_view;
```

**Passo 2 — reparar a(s) migration(ões) confirmadas**, uma por vez ou em lote (aceita múltiplos timestamps):

```bash
supabase migration repair <timestamp1> [<timestamp2> ...] --status applied
```

**Passo 3 — validar** que ficou correto:

```bash
supabase migration list
```

As colunas `Local` e `Remote` devem bater para os timestamps reparados.

**Aplicado nesta story (2026-07-22):**
```bash
supabase migration repair 20260722020000 --status applied --yes
# Repaired migration history: [20260722020000] => applied
```
Confirmado com `supabase migration list` — `Local` e `Remote` batendo para `20260722020000`.

**Pendência separada, fora do escopo desta story:** as 41 migrations anteriores (`20260419_02_add_asaas_fields.sql` até `20260722010003_competitor_monitoring.sql`, listadas via `supabase db push --dry-run --include-all`) seguem com o mesmo desalinhamento. Usar este mesmo runbook para resolvê-las, migration por migration, confirmando cada uma antes de reparar — não reparar em lote sem verificação, pelo risco descrito no Passo 1.
