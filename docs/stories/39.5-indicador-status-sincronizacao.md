# Story 39.5: Indicador de Status de Sincronização

**Epic:** 39 (Evolução do Dashboard — Analytics do Gestor)
**Story ID:** 39.5
**Status:** Ready (PO GO 10/10)
**Story Points:** 3
**Priority:** P2 — Grupo D (sem dependência, pode rodar a qualquer momento)
**Depende de:** Nenhuma story desta epic

## User Story

Como **gestor**, quero ver, direto no dashboard, se a última sincronização de calendários deu certo ou falhou e quando ela ocorreu, para que eu perceba um problema de sync sem precisar entrar na tela de Sincronização para descobrir.

---

## Description

Adicionar um texto de status logo abaixo do botão "Sincronizar" no cabeçalho do dashboard (`src/app/[locale]/dashboard/page.tsx`, linhas ~458-464): verde + "Sincronizado às DD/MM HH:MM" em caso de sucesso, vermelho + "Falha na sincronização às DD/MM HH:MM" em caso de erro.

**Achado importante da investigação de codebase — muda o escopo real desta story:** a tabela `sync_logs` (já existe, `supabase/migrations/20260100000000_bootstrap_core_tables.sql`, colunas `status`, `error_message`, `synced_at`, `sync_type`, `direction`, `property_listing_id`) parecia ser a fonte óbvia para este indicador, mas **não é escrita pelo fluxo de sync inbound** (`src/app/api/cron/sync-ical/route.ts`, `src/app/api/sync/import/route.ts`) — só é usada por `src/lib/reservations/syncToOutboundPlatforms.ts` (direção oposta, Lodgra → plataformas). O único sinal hoje é `property_listings.last_synced_at`, atualizado em `sync-ical/route.ts:403` **apenas no caminho de sucesso** (fim da função `syncOneListing`) — não existe nenhum registro persistido de horário/motivo de falha.

**Portanto, esta story precisa, além do componente de UI:**
1. Adicionar `INSERT` em `sync_logs` (sucesso e falha) nos pontos onde `cron/sync-ical/route.ts` e `sync/import/route.ts` hoje só atualizam `last_synced_at` ou fazem `console.log`/`console.error` sem persistir. Usar `direction: 'inbound'`, `sync_type: 'ical'`, `status: 'success' | 'failed'`, `error_message` preenchido no caso de falha.
2. O indicador no dashboard lê a linha mais recente de `sync_logs` (maior `synced_at`) para as propriedades da organização — via join `sync_logs.property_listing_id → property_listings.property_id → properties.organization_id` — e mostra verde/vermelho conforme `status` dessa linha mais recente.

## Acceptance Criteria

- [ ] `cron/sync-ical/route.ts`: grava em `sync_logs` tanto no caminho de sucesso (por listing sincronizado) quanto no de falha (erro capturado no `catch`, hoje só logado no console).
- [ ] `sync/import/route.ts`: mesma coisa (gravar sucesso e falha em `sync_logs`).
- [ ] Indicador abaixo do botão "Sincronizar": texto verde + "Sincronizado às DD/MM HH:MM" quando a sincronização mais recente da organização teve `status = 'success'`.
- [ ] Texto vermelho + "Falha na sincronização às DD/MM HH:MM" quando a mais recente teve `status = 'failed'`.
- [ ] Se não houver nenhum registro de sync ainda (organização nova, nenhum iCal configurado), não mostrar nem verde nem vermelho — omitir o indicador ou mostrar um terceiro estado neutro (ex.: "Nenhuma sincronização ainda"), nunca inventar um status.
- [ ] Formato de data/hora `DD/MM HH:MM`, fuso horário local do usuário (mesmo padrão já usado em outras partes do dashboard).
- [ ] Design segue `docs/design-system/design.md` (cores semânticas: verde sucesso, vermelho erro).

## Scope

**IN:**
- Gravação de `sync_logs` para sync inbound (iCal cron + import manual)
- Indicador visual verde/vermelho + timestamp no dashboard

**OUT:**
- Alertas no sino de notificações para falha de sync — **isso é da Story 39.6** (o sino cobre "falha de sync" como um dos gatilhos, mas o indicador textual abaixo do botão é desta story; não duplicar a mesma lógica de leitura, extrair para uma função compartilhada se fizer sentido quando a 39.6 for implementada)
- Sincronização de outras integrações (Google Merchant, WhatsApp) — fora de escopo, esta story é só sobre o sync de calendários/reservas que o botão "Sincronizar" dispara

## Risks

| Risk | Mitigation |
|------|-----------|
| Volume alto de linhas em `sync_logs` ao longo do tempo (um insert por listing por execução do cron) | Índice já existiria implicitamente por PK; se necessário, `CREATE INDEX` em `(property_listing_id, synced_at DESC)` — avaliar se a tabela já tem esse índice antes de criar outro |
| Falha silenciosa não capturada pelo `try/catch` atual (algum erro que já escapa antes de chegar ao catch existente) | Revisar os pontos de `try/catch` atuais nos dois arquivos antes de adicionar o insert — garantir que cobre o caso real de falha, não só o caminho feliz |

---

## Technical Notes

- `sync_logs.direction` já existe como coluna livre (VARCHAR, sem enum/check constraint visível na migration original) — usar `'inbound'` como convenção, documentar no código.
- Este indicador é por **organização**, não por listing individual — se uma organização tem 5 listings e 4 sincronizam bem mas 1 falha, a política mais simples e mais útil para o gestor é mostrar vermelho (qualquer falha recente é motivo de atenção) — confirmar esse entendimento com Fabio se parecer contra-intuitivo ao implementar.

---

## Notes for Developer

1. Esta é a story mais simples da epic e não depende de nada — bom candidato para rodar em paralelo desde o início, inclusive por um agente diferente das demais.
2. Ao adicionar o insert em `sync_logs`, não remover a atualização de `last_synced_at` em `property_listings` — outras partes do sistema (`/sync/page.tsx`) já dependem dela.
