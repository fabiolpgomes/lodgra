## User Story

Como **gestor**, quero ver, direto no dashboard, se a última sincronização de calendários deu certo ou falhou e quando ela ocorreu, para que eu perceba um problema de sync sem precisar entrar na tela de Sincronização para descobrir.

---

## Description

Adicionar um texto de status logo abaixo do botão "Sincronizar" no cabeçalho do dashboard (`src/app/[locale]/dashboard/page.tsx`): verde + "Sincronizado às DD/MM HH:MM" em caso de sucesso, vermelho + "Falha na sincronização às DD/MM HH:MM" em caso de erro.

**Achado importante da investigação de codebase — muda o escopo real desta story:** a tabela `sync_logs` (já existe, `supabase/migrations/20260100000000_bootstrap_core_tables.sql`, colunas `status`, `error_message`, `synced_at`, `sync_type`, `direction`, `property_listing_id`) parecia ser a fonte óbvia para este indicador, mas **não é escrita pelo fluxo de sync inbound** (`src/app/api/cron/sync-ical/route.ts`, `src/app/api/sync/import/route.ts`) — só é usada por `src/lib/reservations/syncToOutboundPlatforms.ts` (direção oposta, Lodgra → plataformas). O único sinal antes desta story era `property_listings.last_synced_at`, atualizado apenas no caminho de sucesso — não existia nenhum registro persistido de horário/motivo de falha. **Confirmado pelo `@dev` durante a implementação** (ver Dev Notes).

**Portanto, esta story precisou, além do componente de UI:**
1. Adicionar `INSERT` em `sync_logs` (sucesso e falha) em `cron/sync-ical/route.ts` e `sync/import/route.ts` (ambos os modos deste último — `property_ids` em lote e o modo legado `url + property_id + listing_id`).
2. O indicador no dashboard lê a linha mais recente de `sync_logs` para as propriedades da organização e mostra verde/vermelho conforme o `status` dessa linha.

## Dev Notes (implementação — modo YOLO)

**Nota de processo:** este arquivo não existia no worktree isolado onde a implementação rodou (a story só tinha sido commitada em `main` depois do worktree já ter sido criado). O `@dev` reconstruiu a story a partir do briefing técnico recebido, implementou, e o arquivo foi reconciliado com a versão original (esta) durante o merge — mantendo a estrutura/validação `@po` originais e incorporando as decisões técnicas reais abaixo.

### Decisões técnicas
1. **`status: 'failed'`** (não `'error'`) — a coluna real é `VARCHAR` livre sem `CHECK` constraint; `SyncLog.status` (TS) estendido para `'success' | 'failed' | 'error' | 'partial' | 'pending'` para cobrir valores legados (`syncToOutboundPlatforms.ts` já usa `'pending'`) sem quebrar código existente.
2. **Sucesso/falha é por listing, não por evento** — um listing é `'success'` sempre que a função de sync completa sem lançar exceção (mesmo com eventos individuais `skipped`, ex. sobreposição de datas), `'failed'` só quando lança exceção. Consistente com o comportamento pré-existente de `last_synced_at`.
3. **Insert em `sync_logs` é fire-and-forget resiliente** — se o próprio INSERT falhar, é só logado (`console.warn`), nunca interrompe o sync real.
4. **Modo legado de `/api/sync/import`** — antes, uma exceção nesse modo caía num catch genérico sem persistir nada e sem contexto do listing; agora tem catch dedicado, grava o log corretamente e retorna mensagem de erro mais específica (não muda o status code HTTP, não quebra consumidores existentes).
5. **Query do dashboard** reaproveita o padrão idiomático já usado no arquivo (`.in('property_listings.property_id', propertyIds)`), mesmo estilo da query de `reservations`/`upcomingCheckIns`.
