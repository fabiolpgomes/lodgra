## Story

**Como** gestor de propriedades,
**Quero** definir preços diferentes por época e mínimo de noites,
**Para que** os hóspedes vejam o preço correcto em cada período e as reservas directas sejam rentáveis.

---

## Contexto de Negócio

Preços fixos perdem receita em época alta e dificultam preenchimento em época baixa. O motor de preços mínimo viável permite:
- Preço base (já existe em `properties.base_price`)
- Regras por período (época alta, Natal, Verão)
- Mínimo de noites por período

Sem pricing rules configuradas, o sistema usa sempre o `base_price` da propriedade.

## Dev Notes

### Schema — Migration
```sql
-- supabase/migrations/20260320_02_pricing_rules.sql
CREATE TABLE IF NOT EXISTS pricing_rules (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  property_id UUID NOT NULL REFERENCES properties(id) ON DELETE CASCADE,
  organization_id UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
  name TEXT NOT NULL,                          -- ex: "Época Alta Verão 2026"
  start_date DATE NOT NULL,
  end_date DATE NOT NULL,
  price_per_night NUMERIC(10,2) NOT NULL,
  min_nights INTEGER NOT NULL DEFAULT 1,
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  CONSTRAINT valid_date_range CHECK (end_date >= start_date)
);

ALTER TABLE pricing_rules ENABLE ROW LEVEL SECURITY;

-- Leitura pública para propriedades públicas
CREATE POLICY "Public read for public properties"
  ON pricing_rules FOR SELECT
  USING (
    EXISTS (SELECT 1 FROM properties WHERE id = property_id AND is_public = true)
  );

-- Escrita apenas admin/manager da organização
CREATE POLICY "Org admin/manager write"
  ON pricing_rules FOR ALL
  USING (
    organization_id = get_user_organization_id() AND
    EXISTS (SELECT 1 FROM user_profiles WHERE id = auth.uid() AND role IN ('admin', 'manager'))
  );

CREATE INDEX idx_pricing_rules_property_id ON pricing_rules(property_id);
CREATE INDEX idx_pricing_rules_dates ON pricing_rules(property_id, start_date, end_date);
```
