## Story

**Como** gestor,
**Quero** garantir que reservas não possam ser duplicadas devido a race condition no validação de disponibilidade,
**Para que** o sistema de bookings directos seja confiável e impeça overbooking após pagamento.

---

## Contexto de Negócio

Actualmente, existe uma race condition crítica no sistema de bookings públicos. A validação de disponibilidade apenas conta reservas com status `'confirmed'`, ignorando reservas com status `'pending_payment'`. Isto permite que dois clientes comprem datas identicamente sobrepostas, contanto que ambos completem o pagamento rapidamente.

**Cenário de ataque:**
1. Cliente A selecciona 10-15 Março → verificação de disponibilidade retorna OK (sem reservas confirmas nessas datas)
2. Cliente B selecciona 12-17 Março (sobreposição) → idem, retorna OK
3. Ambos completam pagamento quase em simultâneo → ambas as reservas são confirmadas
4. Propriedade fica dupla-booking

**Root cause:** A query de disponibilidade em `src/app/api/public/properties/[slug]/availability/route.ts` (linha 82) e a validação de conflitos em `src/app/api/public/bookings/route.ts` (linha 124) apenas verificam:
```sql
WHERE status = 'confirmed'
```

Devem incluir também status `'pending_payment'`:
```sql
WHERE status IN ('confirmed', 'pending_payment')
```

**3. Database Migration: `supabase/migrations/20260330_02_fix_reservation_conflict_check.sql`**

Criar trigger que rejeita operações INSERT/UPDATE para reservas que conflituam com reservas `pending_payment`:

```sql
-- Create or replace trigger to validate booking dates
CREATE OR REPLACE FUNCTION validate_booking_dates()
RETURNS TRIGGER AS $$
DECLARE
  conflict_count INTEGER;
BEGIN
  -- Check if new/updated booking conflicts with existing pending_payment or confirmed reservations
  SELECT COUNT(*)
  INTO conflict_count
  FROM reservations
  WHERE property_id = NEW.property_id
    AND id != NEW.id  -- Exclude self (for UPDATE operations)
    AND status IN ('confirmed', 'pending_payment')
    AND (
      (NEW.check_in < check_out AND NEW.check_out > check_in)  -- Overlap check
    );

  IF conflict_count > 0 THEN
    RAISE EXCEPTION 'Booking dates conflict with existing reservation';
  END IF;

  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Drop existing trigger if it exists (to avoid errors on re-runs)
DROP TRIGGER IF EXISTS validate_booking_dates_trigger ON reservations;

-- Create new trigger
CREATE TRIGGER validate_booking_dates_trigger
BEFORE INSERT OR UPDATE ON reservations
FOR EACH ROW
EXECUTE FUNCTION validate_booking_dates();
```

### Testing Strategy

**Unit Tests** (`src/app/api/public/__tests__/availability.test.ts` e booking conflict):

1. **Test: Availability API excludes pending_payment**
   - Setup: Create reservation with status `pending_payment` for dates 10-15 March
   - Request: GET `/api/public/properties/slug/availability`
   - Assert: Response includes 10-15 March as unavailable

2. **Test: Booking creation fails if pending_payment conflict**
   - Setup: Create reservation with status `pending_payment` for 10-15 March
   - Request: POST `/api/public/bookings` para 12-17 March (overlap)
   - Assert: Response is 409 Conflict with error message
   - Assert: No new reservation created in DB

3. **Test: Booking creation succeeds if no pending/confirmed conflict**
   - Setup: Create reservation with status `pending_payment` para 10-15 March
   - Request: POST `/api/public/bookings` para 20-25 March (no overlap)
   - Assert: Response is 200 OK
   - Assert: New reservation created with status `pending_payment`

4. **Test: Idempotency — second booking after first paid**
   - Setup: Two simultaneous requests for same dates (simulate race condition)
   - Execution: First request succeeds, second request should fail (409)
   - Assert: Only one reservation in DB for those dates

**Manual Testing**:
1. Open two browser windows
2. Select same dates on property public page
3. Proceed to payment for both
4. Complete payment on both quickly
5. Assert: Only one reservation is confirmed (other should error or be cancelled)
