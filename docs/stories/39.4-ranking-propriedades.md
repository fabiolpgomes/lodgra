## User Story

Como **gestor**, quero ver quais propriedades têm o melhor e o pior desempenho por RevPAR, para que eu saiba onde investir atenção (revisar preço, ocupação, ou até reconsiderar a parceria com o proprietário) sem precisar abrir cada propriedade individualmente.

---

## Description

Novo card "Ranking de Propriedades" — top 3 e bottom 3 propriedades por RevPAR do mês corrente, usando `monthly_property_metrics` (criada na Story 39.1) como fonte:

```
ADR (por propriedade) = gross_revenue / nights_sold
RevPAR (por propriedade) = ADR × (nights_sold / available_nights)
```

Nota: a Story 39.2 calcula RevPAR agregado (org-wide) como `ADR × currentMonthOccupancy` (que já vem calculado na página). Esta story precisa do RevPAR **por propriedade individual**, então usa `nights_sold / available_nights` da própria linha de `monthly_property_metrics` daquela propriedade — **mesma fórmula, aplicada por linha em vez de agregada**. Reaproveitar a mesma função de cálculo se a Story 39.2 já tiver extraído uma (ver Technical Notes da 39.2).

- Excluir propriedades com `booking_count = 0` ou dado insuficiente no mês (não faz sentido rankear por RevPAR uma propriedade sem nenhuma reserva — isso não é "pior desempenho", é "sem dado") — mostrar separadamente se fizer sentido (ex.: "3 propriedades sem reservas este mês"), não misturar no ranking numérico.
- Se houver menos de 6 propriedades no total (menos que top 3 + bottom 3), ajustar a exibição (não duplicar a mesma propriedade nas duas listas).

### 2. Decisões de design da função de ranking

- **Elegibilidade para ranking numérico:** uma propriedade só entra em `top`/`bottom` se tiver uma linha em `monthly_property_metrics` para o mês corrente **e** `booking_count > 0` **e** `nights_sold > 0`. Qualquer uma dessas condições falhando joga a propriedade para `withoutBookings` (exibido como texto no rodapé do card, não misturado ao ranking numérico) — implementa literalmente o AC "não entram no ranking numérico — tratamento visual separado".
- **Divisão balanceada para N < 6 propriedades rankáveis** (`splitTopBottomCounts`): em vez de sempre tentar "top 3 primeiro, resto no bottom" (o que geraria splits like 3/1 para N=4), a divisão é balanceada: `topCount = min(3, ceil(N/2))`, `bottomCount = min(3, N - topCount)`. Resultado: N=1→1/0, N=2→1/1, N=3→2/1, N=4→2/2, N=5→3/2, N≥6→3/3. Nunca há sobreposição de propriedade entre as duas listas. Este é o único ponto onde fiz uma escolha de design não 100% ditada pela story (que só exige "não duplicar") — documentando aqui para revisão do `@po`/`@qa` caso um split diferente (ex.: sempre priorizar 3 no top) seja preferido.
- **Empate de RevPAR:** desempate determinístico por `propertyName` (ordem alfabética), com `propertyId` como critério final — garante que a ordem não mude entre renders/refreshes sem motivo (relevante porque `Array.prototype.sort` é estável, mas isso sozinho não desempata valores iguais sem um critério secundário explícito).
- **Filtro de propriedade única:** quando `selectedPropertyId` está definido (usuário filtrou para 1 propriedade específica no dropdown do topo), o card **não busca `monthly_property_metrics`** (evita query desnecessária) e renderiza uma mensagem explicando que o ranking precisa de "Todas as propriedades" selecionadas — escolha de "mostrar mensagem apropriada" em vez de "ocultar completamente", ambas permitidas pelo AC; mensagem foi preferida para não dar a impressão de que o card "sumiu" por erro.

### 3. Query de `monthly_property_metrics`

- Filtrada por `organization_id` explicitamente (a view é materialized view sem RLS — mesma ressalva documentada na Story 39.1 — controle de acesso é responsabilidade de quem consome, e este é o primeiro consumo real da view no código de produção).
- Filtrada por `metric_month = '{ano}-{mês}-01'` (primeiro dia do mês corrente, formato `DATE`), compatível com `DATE_TRUNC('month', check_in)::date` usado na migration da 39.1.
- Busca **todas** as propriedades ativas da organização (`allProperties`, não `propertyIds` filtrado), porque o ranking precisa comparar propriedades entre si — só é pulada inteiramente quando há filtro de propriedade única (ver acima).
