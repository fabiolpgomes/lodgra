# Story 31.2: Dynamic Company Branding (Logos, Colors, Favicons)

**Epic:** Epic 31 — Multi-Tenant Infrastructure (Branded Direct Booking)  
**Status:** Done  
**Priority:** High  
**Type:** Backend + Frontend  
**Assigned to:** @dev  
**Complexity:** Medium (T-shirt: M)

---

## Story

**As** a Lodgra product owner,  
**I want** each company to customize their booking page branding (logo, colors, favicon),  
**So that** the booking page reflects each company's brand identity and improves conversion through brand recognition.

---

## Context

With wildcard DNS and Vercel configured (Story 31.1), subdomains now route to the app. However, all companies see the generic Lodgra branding. Phase 2 introduces company-level customization, starting with visual branding (logos, colors, favicons).

**Current State:**
- Middleware extracts org_id from subdomain
- Booking page reads x-org-slug header
- Database supports organization_id isolation
- No branding customization yet

**Desired State:**
- Companies can upload logo and favicon
- Companies can customize primary brand colors
- Booking page dynamically applies company branding
- Email confirmations use company branding

## Acceptance Criteria

### AC1: Database Schema for Company Branding
**Given:** An organization exists in Supabase  
**When:** Admin uploads branding data  
**Then:**
- [ ] `organization_branding` table created with fields:
  - `id` (UUID primary key)
  - `organization_id` (FK to organizations)
  - `logo_url` (text, nullable — uploaded to Vercel Blob Storage)
  - `favicon_url` (text, nullable)
  - `primary_color` (text, hex color code, default: Lodgra blue #1E40AF)
  - `secondary_color` (text, hex, default: Lodgra gray #6B7280)
  - `accent_color` (text, hex, default: Lodgra gold #FFC000)
  - `created_at` (timestamp)
  - `updated_at` (timestamp)
- [ ] RLS (Row Level Security) enforced: users can only see branding for their organization
- [ ] Unique constraint on organization_id (one branding config per org)

### AC2: File Upload Endpoint
**Given:** Admin is logged in with org ownership  
**When:** POST request to `/api/organizations/[orgId]/branding/upload`  
**Then:**
- [ ] Accepts file uploads (logo PNG/JPG max 2MB, favicon ICO/PNG max 100KB)
- [ ] Uploads files to Vercel Blob Storage with org-namespaced paths
- [ ] Returns signed URLs for frontend to consume
- [ ] Returns 401 if user doesn't own the organization
- [ ] Returns 400 if file exceeds size limits or wrong format

### AC3: Branding Colors API Endpoint
**Given:** Admin wants to update brand colors  
**When:** PATCH request to `/api/organizations/[orgId]/branding/colors`  
**Then:**
- [ ] Accepts JSON: `{ primary_color, secondary_color, accent_color }`
- [ ] Validates hex color format (e.g., #FFFFFF)
- [ ] Updates database and returns updated branding object
- [ ] Returns 401 if user doesn't own organization
- [ ] Returns 400 if invalid color format

### AC4: Branding Fetch API
**Given:** Booking page is loading for organization  
**When:** GET request to `/api/organizations/[orgSlug]/branding`  
**Then:**
- [ ] Returns branding object for the organization (or defaults if not set)
- [ ] Response includes: logo_url, favicon_url, primary_color, secondary_color, accent_color
- [ ] Works for non-authenticated requests (public read)
- [ ] Returns 404 if organization not found

### AC5: Booking Page Dynamic Branding
**Given:** Company branding is stored in database  
**When:** Booking page loads (e.g., `pousada.lodgra.io/booking`)  
**Then:**
- [ ] Fetches branding from `/api/organizations/[orgSlug]/branding`
- [ ] Applies dynamic styles:
  - Primary color used for CTAs, buttons, headers
  - Secondary color used for card borders, dividers
  - Accent color used for highlights (price badges, badges)
- [ ] Renders company logo in header (if provided)
- [ ] Sets favicon to company favicon (if provided)
- [ ] Falls back to Lodgra defaults if branding not configured
- [ ] No visual regression if branding missing

### AC6: Favicon Dynamic Application
**Given:** Company has uploaded favicon  
**When:** Browser loads `pousada.lodgra.io/booking`  
**Then:**
- [ ] Dynamic favicon endpoint serves company favicon: `/api/favicon?orgSlug=[slug]`
- [ ] Favicon is cached for 24 hours (Cache-Control: public, max-age=86400)
- [ ] Falls back to Lodgra favicon if company favicon not set
- [ ] Works across all pages (not just /booking)

### AC7: Admin Dashboard for Branding Management
**Given:** Admin navigates to organization settings  
**When:** Admin visits `/settings/organizations/[orgId]/branding`  
**Then:**
- [ ] Form displays current branding (logo, favicon, colors)

- [ ] Logo upload input (drag-and-drop supported)
- [ ] Favicon upload input
- [ ] "Save" button updates branding in database
- [ ] "Preview" button shows how booking page will look

- [ ] Error handling if upload fails

- [ ] RLS policies prevent cross-org branding data leakage

## Dev Notes

### Technical Approach

**1. Database Migration (Supabase)**
```sql
CREATE TABLE organization_branding (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  organization_id UUID NOT NULL UNIQUE REFERENCES organizations(id) ON DELETE CASCADE,
  logo_url TEXT,
  favicon_url TEXT,
  primary_color TEXT DEFAULT '#1E40AF',
  secondary_color TEXT DEFAULT '#6B7280',
  accent_color TEXT DEFAULT '#FFC000',
  created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
  updated_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

ALTER TABLE organization_branding ENABLE ROW LEVEL SECURITY;

CREATE POLICY org_branding_select ON organization_branding
  FOR SELECT USING (
    auth.uid() IN (
      SELECT user_id FROM organization_members 
      WHERE organization_id = organization_branding.organization_id
    )
  );

CREATE POLICY org_branding_update ON organization_branding
  FOR UPDATE USING (
    auth.uid() IN (
      SELECT user_id FROM organization_members 
      WHERE organization_id = organization_branding.organization_id 
      AND role = 'admin'
    )
  );
```

**2. File Upload Strategy**
- Use Vercel Blob Storage for logo/favicon
- Path format: `orgs/{orgId}/logo` and `orgs/{orgId}/favicon`
- Return signed URLs from upload endpoint
- Cache headers: 1 year for immutable assets

**3. Frontend Implementation**
- Create `<BrandingProvider>` context to pass branding to components
- Update `BookingPageClient` to use branding colors (Tailwind CSS variables)
- Dynamic favicon: use HTML `<link rel="icon">` meta tag in RSC

**4. Color Application**
- Use CSS custom properties (--color-primary, --color-secondary, --color-accent)
- Apply in booking page layout
- Update button, badge, and header styles

## Risks & Mitigations

### CRITICAL

**R1: SVG Injection Attack via Logo Upload**
- **Severity:** CRITICAL
- **Probability:** High
- **Impact:** Code execution, session hijacking, XSS
- **Description:** SVG files can contain embedded scripts. If uploaded SVG is returned with `Content-Type: image/svg+xml`, browser executes embedded JavaScript.
- **Mitigation:** 
  - Sanitize SVG before serving (use DOMPurify or similar)
  - Return all image files with `Content-Type: application/octet-stream` (force download)
  - Restrict uploads to PNG/JPG only (reject SVG at upload boundary)
  - Verify image dimensions/integrity server-side using Sharp or ImageMagick
- **Checklist:** 
  - [ ] Upload validation rejects SVG files
  - [ ] PNG/JPG returned with correct Content-Type header
  - [ ] Test SVG with embedded script is rejected or sanitized

**R2: RLS Policy Vulnerability - Cross-Organization Data Leak**
- **Severity:** CRITICAL
- **Probability:** Medium (if RLS misconfigured)
- **Impact:** Data exposure, competitor intelligence, regulatory violation
- **Description:** If RLS policies on `organization_branding` table are not properly scoped to `auth.uid()` → `organization_members`, queries from one org could expose another org's branding (logo URLs, colors).
- **Mitigation:**
  - RLS policy MUST check: `auth.uid() IN (SELECT user_id FROM organization_members WHERE organization_id = organization_branding.organization_id)`
  - Test RLS with multi-tenant setup (Org A user cannot read Org B branding)
  - Audit all API endpoints to ensure org_id parameter matches authenticated user's orgs
  - Use `@po` to validate RLS during story review
- **Checklist:**
  - [ ] RLS policy explicitly limits to authenticated user's orgs
  - [ ] Multi-tenant test: Org A user denied access to Org B branding
  - [ ] API endpoints validate org_id ownership before returning data

### HIGH

**R3: Denial of Service via File Upload**
- **Severity:** HIGH
- **Probability:** Medium
- **Impact:** Service degradation, storage costs, availability
- **Description:** Admin could upload extremely large files (100MB+) or many concurrent uploads, exhausting storage quota and rate limits.
- **Mitigation:**
  - Enforce strict file size limits: logo max 2MB, favicon max 100KB
  - Rate limit: 1 upload per 10 seconds per org
  - Reject files if total org storage > quota (e.g., 50MB)
  - Use Vercel Blob Storage's built-in rate limiting
- **Checklist:**
  - [ ] File size validation enforced at upload endpoint
  - [ ] Rate limiting test: rapid uploads rejected with 429 Too Many Requests
  - [ ] Storage quota test: oversized total rejected

### MEDIUM

**R4: Color Contrast & Accessibility**
- **Severity:** MEDIUM
- **Probability:** Medium
- **Impact:** WCAG violation, accessibility failure, poor UX for color-blind users
- **Description:** Admin sets primary_color to "#FFFF00" (yellow), making white text unreadable (0.37:1 contrast ratio).
- **Mitigation:**
  - Validate color contrast ratio when color is set (WCAG AA minimum 4.5:1)
  - Provide visual feedback: "This color combination fails accessibility. Try lighter/darker shade."
  - Pre-compute fallback colors (darker/lighter variant) if contrast fails
  - Default fallback: if primary_color fails contrast, use darker variant automatically
- **Checklist:**
  - [ ] Color validation checks contrast ratio on PATCH
  - [ ] Admin dashboard shows contrast warning (red/green indicator)
  - [ ] Fallback color applied if necessary

**R5: Favicon Cache Invalidation**
- **Severity:** MEDIUM
- **Probability:** Low (but high impact when occurs)
- **Impact:** Stale favicon cached in browsers, brand confusion
- **Description:** Browser caches favicon aggressively. If admin uploads new favicon, user's browser shows old favicon for weeks due to 1-year Cache-Control header.
- **Mitigation:**
  - Include favicon URL as dynamic endpoint: `/api/favicon?orgSlug={slug}&t={hash}`
  - Hash = hash(favicon_url), changes when favicon_url changes
  - Cache-Control: public, max-age=86400 (24 hours, not 1 year)
  - Provide "Clear cache" button in admin dashboard
- **Checklist:**
  - [ ] Favicon endpoint includes cache buster (hash of favicon_url)
  - [ ] Cache-Control set to 24 hours, not indefinite
  - [ ] Test: upload new favicon, verify old version drops within 24h

**R6: Storage Cost Accumulation**
- **Severity:** MEDIUM
- **Probability:** Medium
- **Impact:** Unexpected cost increase, financial exposure
- **Description:** Over 2 years, if 10k companies each upload 2MB logo + 100KB favicon, total = 20GB, adding ~$500/month to Vercel Blob costs.
- **Mitigation:**
  - Monitor storage usage per org (quota soft limit)
  - Document storage limits in admin dashboard: "Current: 5MB / 50MB quota"
  - Implement storage cleanup: delete old versions when new image uploaded
  - Alert Ops if any org exceeds quota
- **Checklist:**
  - [ ] Storage quota enforced (soft limit with warning, hard limit blocks)
  - [ ] Admin can see current usage in dashboard
  - [ ] Old image versions cleaned up on new upload

**R7: Performance - Dynamic Color Calculation**
- **Severity:** MEDIUM
- **Probability:** Low
- **Impact:** Slow page loads, poor user experience
- **Description:** If colors are calculated/validated on every request (contrast check, brightness calculation), homepage load time increases.
- **Mitigation:**
  - Pre-compute contrast ratio at color save time (PATCH endpoint), not on page load
  - Cache branding object for 5 minutes in app memory
  - Use CSS custom properties (no JavaScript calculation on page load)
  - Query branding once per page load, not per component
- **Checklist:**
  - [ ] Branding data cached (5min+ TTL)
  - [ ] Color validation happens at save time, not page load
  - [ ] CSS variables set server-side (RSC), not client-side
