## Story

**Como** hóspede,
**Quero** poder pagar por uma reserva direta em EUR (Stripe Portugal),
**Para que** o proprietário receba um pagamento automático via SEPA/IBAN e Lodgra receba a comissão.

---

## Contexto de Negócio

Lodgra oferece direct bookings (alternativa ao Airbnb). Quando hóspede faz reserva:
1. Cria payment intent em EUR (Stripe Portugal)
2. Split automático: proprietário recebe 85%, Lodgra 15%
3. Proprietário recebe via SEPA em 2-3 dias úteis
4. Suporta reembolsos (completo e parcial)

Usa Stripe Connect (Express account) para cada proprietário isolado (multi-tenant).

### O que já existe
- Stripe customer criado na signup
- Tabela organizations, bookings
- Auth para proprietários e hóspedes

### O que falta
- Onboarding para Connect (Express account por proprietário)
- Payment intent creation com split automático
- Webhook listeners para charge events
- Payout tracking e status
- Refund flow (UI + API)
- Checkout UI integrada

## Riscos & Mitigação

**Payout Failures & Split Math Errors** — Stripe Connect split (85% owner, 15% Lodgra) pode falhar se destination account suspended ou se split percentages arredondadas incorrectamente (R$100 split em centavos causa erro). **Mitigation:** Testar split logic com 50+ valores (centavos, moedas fracionadas); implementar audit trail de todos os splits em database; monitorar failed payouts em Sentry com alertas imediatos; usar Stripe payout simulation em test mode.

**Fraud Detection & Chargeback Handling** — Hóspedes podem fazer pagamento com cartão roubado, chargebacks podem chegar meses depois. Se refund parcial falha, proprietário fica com saldo incorrecta. **Mitigation:** Implementar Stripe radar (fraud detection automática); documentar fluxo de chargeback (refund automático, email notificações); unit tests para refund partial logic com edge cases (refund > original amount, negative refund, etc.).

**Stripe Connect Account Limits & Onboarding** — Proprietários podem atingir limites Stripe (transaction volume, payout frequency), onboarding pode falhar se IBAN inválido. **Mitigation:** Validar IBAN format client-side antes de onboarding; documentar Stripe Connect límites em onboarding page; implementar webhook para account.updated (captura suspensões); graceful degradation se Connect account não activo.

## Dev Notes

### Connect Onboarding Flow

```typescript
// GET /api/stripe/connect/link
// Generates Express account link (redirect_url)
const accountLink = await stripePT.accountLinks.create({
  account: org.stripe_pt_connect_id || (new Express account),
  type: 'account_onboarding',
  refresh_url: `${baseUrl}/api/stripe/connect/refresh`,
  return_url: `${baseUrl}/dashboard/stripe-connect-success`,
});
```

### Payment Intent with Split

```typescript
// POST /api/booking/{bookingId}/payment
const paymentIntent = await stripePT.paymentIntents.create({
  amount: totalAmount * 100, // EUR cents
  currency: 'eur',
  on_behalf_of: organization.stripe_pt_connect_id,
  application_fee_amount: commission * 100, // Lodgra fee
  transfer_data: {
    destination: organization.stripe_pt_connect_id
  },
  metadata: {
    booking_id: bookingId,
    org_id: organization.id,
  }
});
```

### Webhook Events to Handle

```
charge.succeeded → payment.status = 'succeeded'
charge.failed → payment.status = 'failed'
charge.refunded → payment.status = 'refunded', create refund record
charge.dispute.created → payment.status = 'disputed', alert admin
```

### Refund Logic

```typescript
// Refund: full or partial
const refund = await stripePT.refunds.create({
  charge: payment.stripe_payment_id,
  amount: refundAmount * 100, // optional for partial
  reason: 'requested_by_customer' | 'duplicate' | 'fraudulent'
});

// Update payment record
await updatePayment(paymentId, {
  status: 'refunded',
  refund_amount: refundAmount,
  refund_reason: reason,
});
```
