## Story

**Como** proprietário que lista imóveis no Lodgra,
**Quero** poder registar as coordenadas geográficas (latitude e longitude) de cada imóvel,
**Para que** o Google indexe correctamente o imóvel no Google Vacation Rentals com localização precisa.

---

## Context

O Google Rich Results Test para "Aluguel por temporada" reporta o erro crítico:
> **"O campo 'geo' não foi encontrado"**

O campo `geo` no Schema.org `VacationRental → containsPlace (Accommodation)` requer `GeoCoordinates` com `latitude` e `longitude`. A tabela `properties` na DB não tem estas colunas.

**Decisão de arquitectura — entrada manual (não geocoding automático):**
- Geocoding automático (Google Maps API, Nominatim) implica custo ou rate limits
- Para 7 propriedades activas, entrada manual é mais fiável e imediata
- O dashboard já tem o formulário de edição de propriedades em `src/app/[locale]/properties/[id]/edit/page.tsx`
- Adicionar dois campos `latitude` / `longitude` ao mesmo formulário é suficiente

**Dados de localização que já existem:**
- `address`, `city`, `country`, `postal_code` — já preenchidos e usados no schema
- `containsPlace → Accommodation` já está em produção (Story 24.1)

---

## Acceptance Criteria

- **AC1** — Migração Supabase adiciona `latitude DECIMAL(10,8)` e `longitude DECIMAL(11,8)` à tabela `properties`, ambas nullable
- **AC2** — O formulário de edição de propriedade (`/properties/[id]/edit`) tem dois campos numéricos: "Latitude" e "Longitude", com placeholder de exemplo (`38.7223`, `-9.1393`)
- **AC3** — O `UPDATE` ao guardar a propriedade inclui `latitude` e `longitude`
- **AC4** — O `generatePropertyJsonLd()` inclui `geo: { '@type': 'GeoCoordinates', latitude, longitude }` dentro de `containsPlace` quando ambos os valores existem
- **AC5** — A query da página pública `/p/[slug]` selecciona `latitude` e `longitude` e passa-os ao `generatePropertyJsonLd()`
- **AC6** — O Google Rich Results Test deixa de reportar "O campo 'geo' não foi encontrado" para propriedades com coordenadas preenchidas

### Padrão de formulário existente (replicar)

O formulário em `page.tsx` usa `formData.get('field_name')`. Exemplo do `postal_code` (linhas 173-178):
```typescript
.update({
  // ...
  postal_code: formData.get('postal_code') as string,
})
```

Para `latitude`/`longitude` usar `parseFloat` com fallback null:
```typescript
const lat = formData.get('latitude') ? parseFloat(formData.get('latitude') as string) : null
const lng = formData.get('longitude') ? parseFloat(formData.get('longitude') as string) : null
```

### Coordenadas de exemplo para teste
- Lisboa: `38.7223, -9.1393`
- Armação de Pêra: `37.1015, -8.3518`
- Porto: `41.1496, -8.6109`

Para obter coordenadas de qualquer morada: Google Maps → clique direito sobre o local → copia as coordenadas.

### Schema.org `geo` dentro de `containsPlace`

```json
"containsPlace": {
  "@type": "Accommodation",
  "address": { ... },
  "geo": {
    "@type": "GeoCoordinates",
    "latitude": 37.1015,
    "longitude": -8.3518
  },
  "numberOfRooms": 1
}
```

### Precision das colunas DB

- `DECIMAL(10,8)` para latitude: máx ±99.99999999 (cobre [-90, 90])
- `DECIMAL(11,8)` para longitude: máx ±999.99999999 (cobre [-180, 180])
- 8 casas decimais = precisão ~1mm (mais do que suficiente)
