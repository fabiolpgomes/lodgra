# Story 24.1: Schema.org VacationRental — Markup Completo para Google Vacation Rentals

**Epic:** 24 — Google Vacation Rentals Distribution

## Story

**Como** proprietário que lista imóveis no Lodgra,
**Quero** que as páginas públicas dos imóveis tenham Schema.org `VacationRental` completo e válido,
**Para que** o Google possa indexar e distribuir os meus imóveis no Google Vacation Rentals sem comissões de OTA.

---

## Context

A função `generatePropertyJsonLd()` em `src/lib/seo/jsonld.ts` já existe e usa `@type: VacationRental` (subtipo correcto de `LodgingBusiness` para Google Vacation Rentals). No entanto, tem vários problemas que impedem a validação pelo Google:

1. **`APP_URL` errado** — aponta para `https://www.lodgra.pt` em vez de `https://lodgra.io`
2. **Nonce em falta** — `<script type="application/ld+json">` na página não tem o atributo `nonce`, causando potencial violação CSP
3. **Imagens desactualizadas** — o JSON-LD usa apenas `property.photos[0]` (array legado) em vez das imagens WebP do sistema `property_images`
4. **Amenities em falta** — as amenities estruturadas são carregadas na página mas nunca passadas ao `generatePropertyJsonLd()`
5. **Campos obrigatórios Google em falta** — `@id`, `url`, `numberOfBathroomsTotal`, `occupancy`, `petsAllowed`

**Decisão de tipo Schema.org:**
`VacationRental` (não `LodgingBusiness`) — é o tipo correcto e reconhecido pelo Google Vacation Rentals para propriedades de alojamento temporário de uso privado.

**O que já funciona:**
- Página pública: `/p/[slug]` com URL única ✅
- JSON-LD é renderizado na página (`<script type="application/ld+json">`) ✅
- OpenGraph metadata completo ✅
- Dados carregados: preço base, taxas, check-in/out, quartos, casas de banho ✅

## Acceptance Criteria

- **AC1** — `APP_URL` em `src/lib/seo/jsonld.ts` usa `process.env.NEXT_PUBLIC_APP_URL` com fallback para `https://lodgra.io` (não `www.lodgra.pt`)
- **AC2** — O `<script type="application/ld+json">` na página `/p/[slug]` inclui o atributo `nonce` lido de `headers()`
- **AC3** — O JSON-LD inclui pelo menos uma imagem WebP do sistema `property_images` (quando disponível); fallback para `photos[]` legado
- **AC4** — O JSON-LD inclui `amenityFeature` com as amenities estruturadas carregadas da tabela `property_amenities`
- **AC5** — O JSON-LD inclui `@id`, `url`, `numberOfRooms`, `numberOfBathroomsTotal`, `occupancy.maxValue`
- **AC6** — Validação positiva no [Google Rich Results Test](https://search.google.com/test/rich-results) para pelo menos uma propriedade de teste
- **AC7** — Sem erros de CSP nos browser DevTools após o deploy (o nonce elimina o risco)
- **AC8** — `generateWebsiteJsonLd()` é adicionado ao layout raiz (`src/app/layout.tsx`) para o Website schema

### Padrão de nonce na página (para replicar em `/p/[slug]/page.tsx`)

O layout raiz já usa este padrão (linha 82 de `src/app/layout.tsx`):
```typescript
import { headers } from 'next/headers'
const nonce = (await headers()).get('x-nonce') ?? undefined
```

O nonce vem do middleware (`src/middleware.ts` linha 25) — gerado por request como `Buffer.from(crypto.randomUUID()).toString('base64')` e propagado via header `x-nonce`.

### Padrão de script JSON-LD com nonce

```tsx
<script
  type="application/ld+json"
  nonce={nonce}
  dangerouslySetInnerHTML={{ __html: JSON.stringify(jsonLd) }}
/>
```

### CSP e nonce

O middleware (`src/middleware.ts:25`) gera um nonce por request. O `applySecurityHeaders()` em `src/lib/middleware/security-headers.ts` aplica `script-src 'nonce-{nonce}'`. Qualquer `<script>` inline sem esse nonce será bloqueado. O layout raiz já propaga o nonce correctamente — replicar o mesmo padrão na página `/p/[slug]`.
