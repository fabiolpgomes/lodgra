## Story

**Como** administrador da plataforma,
**Quero** que os dados de consentimento e eliminação sejam isolados por organização,
**Para que** um admin de Org A nunca veja dados de Org B — conforme RGPD Art. 5 (data minimization).

---

## Contexto

A revisão arquitectural identificou que `consent_records` e `deletion_requests` **não têm `organization_id`**. As RLS policies de admin fazem check de role mas não de organização, permitindo cross-tenant data visibility. Adicionalmente:
- A policy UPDATE de `deletion_requests` não tem `WITH CHECK` — user pode alterar `status` directamente
- FK cascading inconsistente: `consent_records` usa SET NULL, `deletion_requests` usa CASCADE (perde audit trail)

## Dev Notes

### Backfill strategy

```sql
-- Backfill organization_id from user_profiles
UPDATE consent_records cr
SET organization_id = (
  SELECT organization_id FROM user_profiles up WHERE up.id = cr.user_id
)
WHERE cr.user_id IS NOT NULL AND cr.organization_id IS NULL;

-- For anonymous records (user_id IS NULL), set default org
UPDATE consent_records
SET organization_id = '00000000-0000-0000-0000-000000000001'
WHERE user_id IS NULL AND organization_id IS NULL;
```

### RLS pattern para admin org-scoped

```sql
CREATE POLICY "consent_records_select_admin_org"
  ON consent_records FOR SELECT
  USING (
    organization_id = (SELECT organization_id FROM user_profiles WHERE id = auth.uid())
    AND EXISTS (SELECT 1 FROM user_profiles WHERE id = auth.uid() AND role = 'admin')
  );
```

### UPDATE WITH CHECK pattern

```sql
CREATE POLICY "deletion_requests_update_own"
  ON deletion_requests FOR UPDATE
  USING (user_id = auth.uid() AND status = 'pending')
  WITH CHECK (status = 'cancelled' AND user_id = auth.uid());
```
