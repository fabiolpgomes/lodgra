# Story 39.3: Card Receita por Canal

**Epic:** 39 (Evolução do Dashboard — Analytics do Gestor)
**Story ID:** 39.3
**Status:** Ready (PO GO 10/10)
**Story Points:** 5
**Priority:** P1 — Grupo B (independente)
**Depende de:** [39.1](39.1-schema-foundation-metrics.md) (Done)

---

## User Story

Como **gestor**, quero ver a receita e a comissão paga por canal de distribuição (Airbnb, Booking, direto, etc.), e ser alertado quando um único canal concentrar demais a receita, para que eu identifique dependência excessiva de uma plataforma antes que isso vire um risco para o negócio.

---

## Description

Novo card "Receita por Canal", posicionado abaixo de "Receita Mensal" (conforme spec-fonte). Agrupa reservas por `booking_source` no mês corrente (ou no período selecionado, se a página vier a suportar seleção de período — hoje é sempre mês corrente, mesmo padrão dos outros cards):

- **% de receita por canal** — `SUM(total_amount) por booking_source / SUM(total_amount) total`, Receita Bruta, mesma fonte de dados de `monthly_property_metrics` ou query direta em `reservations` (avaliar qual é mais simples — a view agrega por propriedade/mês, não por canal, então provavelmente esta story precisa de uma query própria em `reservations`, agrupando por `booking_source` além de `property_id`/mês; **não expandir `monthly_property_metrics` para incluir canal sem necessidade** — se ficar claro que é necessário, documentar a decisão aqui antes de alterar a migration da 39.1).
- **% de reservas por canal** — contagem de reservas por `booking_source` / total.
- **Comissão por canal** — `SUM(commission_amount) por booking_source` (campo já existente desde antes da Story 39.1, ver `commission_summary`). **Não estimar comissão por taxa fixa** — somar o campo real.
- **Alerta de concentração por canal** — se 1 canal > 60% da receita do mês, sinalizar visualmente no próprio card (não é o mesmo alerta de "concentração por propriedade", que é uma métrica diferente — ver spec-fonte, Conflito 1).

Valores de `booking_source` observados no código atual: `manual`, `ical_import`, `booking`, `airbnb`, `direct`, `booking_api`, `email_parse`. Mapear para rótulos amigáveis (ex.: "Reserva Manual", "Booking.com", "Airbnb", "Direto", "Booking API", "Importado via E-mail") — usar `platforms.display_name` quando a reserva tiver `property_listing_id` associado a uma `platforms` via `property_listings`, como fallback para os `booking_source` que não mapeiam 1:1 a uma plataforma cadastrada (ex.: `manual`, `direct`).

## Acceptance Criteria

- [ ] Card "Receita por Canal" renderizado abaixo de "Receita Mensal" no dashboard.
- [ ] % de receita e % de reservas por canal, usando Receita Bruta (`total_amount`, sem deduções).
- [ ] Comissão por canal somada de `reservations.commission_amount` (campo real, não estimativa).
- [ ] Reservas com `booking_source IS NULL` ou `total_amount IS NULL` são excluídas do cálculo e contabilizadas separadamente (ex.: "N reservas sem canal identificado") — nunca tratadas como um canal "desconhecido" com 0 de receita silenciosamente somado a outro canal.
- [ ] Alerta de concentração por canal: limiar configurável (não hardcoded em JSX solto — usar uma constante nomeada, ex. `CHANNEL_CONCENTRATION_THRESHOLD = 60`), disparando quando 1 canal ultrapassa 60% da receita do mês.
- [ ] Filtro de propriedade no topo (`propertyIds`) afeta este card.
- [ ] Valores monetários respeitam o toggle EUR/BRL.
- [ ] Design segue `docs/design-system/design.md` (mesmo padrão visual dos outros cards da página — cores, radius, tipografia).

## Technical Notes

- `platforms.display_name` existe e pode ser usado como fonte de nome amigável quando disponível via `property_listings.platform_id → platforms`; para `booking_source` sem plataforma associada (`manual`, `direct`), usar um mapa estático de labels.

## Notes for Developer

1. Antes de decidir se usa `monthly_property_metrics` ou uma query própria em `reservations`, confirme se a story 39.4 (Ranking) ou 39.6 (Painel de Alertas — concentração por propriedade) também precisam de dado por canal. Se não, uma query própria e simples nesta story evita inflar a migration da 39.1 desnecessariamente.
2. Rode a query de completude de `booking_source` (documentada nos Dev Notes da Story 39.1, seção 4) antes de implementar o alerta de "reservas sem canal" — isso informa se vale a pena um tratamento visual dedicado ou se é um caso raro.
