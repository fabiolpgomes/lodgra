# Story 31.3: Personalized Booking Page Templates

**Epic:** Epic 31 — Multi-Tenant Infrastructure (Branded Direct Booking)  
**Status:** In Progress  
**Priority:** High  
**Type:** Frontend  
**Assigned to:** @dev  
**Complexity:** Medium (T-shirt: M)

---

## Story

**As** a Lodgra product owner,  
**I want** companies to customize their booking page layout and content (headline, description, featured properties),  
**So that** each company can tailor the booking experience to their market and customer segment.

---

## Context

Story 31.2 added visual branding (colors, logos). Story 31.3 extends customization to page content and layout. Companies can now customize:

- Custom headline and subtitle
- Custom description/welcome text
- Featured properties section (curated vs. all)
- CTA button text and color
- Hero section image (optional)

This positions the platform as white-label — each company controls their entire booking page.

---

## Acceptance Criteria

### AC1: Company Template Configuration Table
**Given:** An organization exists  
**When:** Admin configures booking page template  
**Then:**
- [ ] `organization_templates` table created with fields:
  - `id` (UUID primary key)
  - `organization_id` (FK to organizations, unique)
  - `booking_headline` (text, default: "[Company Name] Properties")
  - `booking_subtitle` (text, nullable)
  - `booking_description` (text, markdown, nullable)
  - `featured_property_ids` (UUID array, nullable — if set, only show these)
  - `show_all_properties` (boolean, default: true)
  - `hero_image_url` (text, nullable)
  - `cta_button_text` (text, default: "Book Now")
  - `template_type` (enum: 'standard' | 'luxury' | 'budget', default: 'standard')
  - `created_at` (timestamp)
  - `updated_at` (timestamp)
- [ ] RLS enforced: users can only modify their org's templates
- [ ] Unique constraint on organization_id

### AC2: Template Configuration API
**Given:** Admin is logged in  
**When:** GET/PATCH `/api/organizations/[orgId]/template`  
**Then:**
- [ ] GET returns current template config (or defaults)
- [ ] PATCH accepts JSON with template fields and updates database
- [ ] Validates headline length (max 100 chars)
- [ ] Validates description (max 500 chars, markdown allowed)
- [ ] Validates featured property IDs (must belong to org)
- [ ] Returns 401 if user doesn't own organization
- [ ] Returns 404 if organization not found

### AC3: Template Fetch for Booking Page
**Given:** Booking page loads for company  
**When:** GET `/api/organizations/[orgSlug]/template`  
**Then:**
- [ ] Returns template config for organization
- [ ] If featured_property_ids set, only return those properties
- [ ] If show_all_properties true, return all properties (default behavior)
- [ ] Include branding colors in response (from Story 31.2)
- [ ] Works for unauthenticated requests
- [ ] Returns 404 if organization not found

### AC4: Booking Page Template Rendering
**Given:** Company has customized their template  
**When:** Booking page loads (e.g., `pousada.lodgra.io/booking`)  
**Then:**
- [ ] Headline displays company's custom headline (not Lodgra generic)
- [ ] Subtitle displays (if provided)
- [ ] Hero image displays (if provided)
- [ ] Description renders as markdown below hero
- [ ] Properties list shows featured properties (if configured) or all properties
- [ ] CTA button text uses custom text (if provided)
- [ ] Falls back to Lodgra defaults if template not configured
- [ ] Properties still filtered by organization (no cross-org data leak)

### AC5: Template Type Variants
**Given:** Admin selects template_type  
**When:** Booking page applies template styling  
**Then:**
- [ ] **'standard'**: Default Lodgra layout (cards grid)
- [ ] **'luxury'**: Larger images, premium spacing, white space
- [ ] **'budget'**: Compact list view, minimal images, focus on price
- [ ] Each template type has distinct CSS class applied to root
- [ ] Booking page renders correctly for all 3 types

### AC6: Admin Dashboard for Template Management
**Given:** Admin navigates to org settings  
**When:** Admin visits `/settings/organizations/[orgId]/template`  
**Then:**
- [ ] Form displays current template settings
- [ ] Headline input field (max 100 chars, counter)
- [ ] Subtitle input field
- [ ] Rich text editor for description (markdown preview)
- [ ] Hero image upload (optional)
- [ ] Template type selector (dropdown: standard/luxury/budget)
- [ ] Featured properties multi-select (search by name)
- [ ] Toggle: "Show all properties" vs. "Featured only"
- [ ] CTA button text input
- [ ] Live preview of booking page (right sidebar)
- [ ] "Save" button persists to database
- [ ] Error handling for validation failures

### AC7: Properties Filtering
**Given:** Company has featured_property_ids set to [prop1, prop2, prop3]  
**When:** Booking page renders `/booking` endpoint  
**Then:**
- [ ] API returns only featured properties in correct order
- [ ] Featured count displayed (e.g., "Showing 3 featured properties")
- [ ] Users can still filter by location, price, amenities (within featured set)
- [ ] No properties from other organizations are visible

### AC8: Backward Compatibility
- [ ] Existing organizations without templates use defaults
- [ ] Root domain `lodgra.io/booking` still shows all properties from all orgs (no template)
- [ ] No regression in search/filter functionality

ALTER TABLE organization_templates ENABLE ROW LEVEL SECURITY;

// If featured_property_ids is set and non-empty, filter to those

if (template.featured_property_ids?.length > 0) {

  // Maintain order from featured_property_ids

  );

## Risks & Mitigations

### CRITICAL

**R1: Cross-Organization Property Leakage via featured_property_ids**
- **Severity:** CRITICAL
- **Probability:** High (if not properly validated)
- **Impact:** Data exposure, regulatory violation (GDPR), competitor intelligence
- **Description:** Admin from Org A could set `featured_property_ids` to include properties from Org B by manually crafting API request. Booking page would then display properties they don't own.
- **Mitigation:**
  - Validate every property_id in featured_property_ids MUST belong to the org making the request
  - Query: `SELECT id FROM properties WHERE id = ANY($1) AND organization_id = $2`
  - Reject with 403 if any property_id doesn't match org
  - Audit logs: log all attempts to set properties from other orgs
- **Checklist:**
  - [ ] PATCH endpoint validates each property_id belongs to org
  - [ ] Multi-tenant security test: Org A cannot include Org B's properties
  - [ ] Audit logs record all invalid property attempts
  - [ ] Error message doesn't leak which properties exist in other orgs

### HIGH

**R2: N+1 Query Problem in Properties Endpoint**
- **Severity:** HIGH
- **Probability:** High
- **Impact:** Database performance degradation, slow page loads
- **Description:** If properties endpoint queries property details in a loop (1 query for template + N queries for each property), GET `/api/organizations/[orgSlug]/template` becomes slow with 100+ properties.
- **Mitigation:**
  - Use single query with JOIN or batch fetch: `SELECT * FROM properties WHERE id = ANY(featured_property_ids)`
  - If ordering required, order in application layer using JavaScript Array.sort()
  - Index `properties(organization_id, id)` for fast filtering
- **Checklist:**
  - [ ] Properties endpoint verified to use batch query, not loop
  - [ ] Load test: 500+ properties returns in < 200ms
  - [ ] Database query profiling shows single query, not N+1

**R3: XSS Vulnerability in Markdown Description Rendering**
- **Severity:** HIGH
- **Probability:** Medium
- **Impact:** JavaScript injection, session hijacking, form hijacking
- **Description:** If `booking_description` markdown is rendered without sanitization, attacker could inject `<img onerror="alert('xss')">` or `<script>` tags.
- **Mitigation:**
  - Use safe markdown library: `remark` + `remark-html` + `rehype-sanitize`
  - Never use `dangerouslySetInnerHTML` without sanitization
  - Strip all HTML tags, render only safe markdown constructs (bold, italic, links)
  - Content Security Policy header: `script-src 'self'` (blocks inline scripts)
- **Checklist:**
  - [ ] Description markdown rendered with sanitization library
  - [ ] Test XSS payloads in description: `<script>`, `onerror=`, `onclick=` all rejected
  - [ ] CSP header configured on booking page

### MEDIUM

**R4: Featured Properties Ordering Not Preserved**
- **Severity:** MEDIUM
- **Probability:** Medium
- **Impact:** UX confusion, admin intention not reflected
- **Description:** If featured_property_ids = [prop3, prop1, prop2], but API returns [prop1, prop2, prop3] (alphabetical), admin's order is lost.
- **Mitigation:**
  - Order properties in application layer using featured_property_ids array order
  - `properties.sort((a, b) => featured_property_ids.indexOf(a.id) - featured_property_ids.indexOf(b.id))`
  - Store order as JSON array in database, not a range
- **Checklist:**
  - [ ] Properties returned in same order as featured_property_ids
  - [ ] Test: set [prop3, prop1, prop2] → verify response order matches

**R5: CSS Specificity Conflicts with Branding Colors**
- **Severity:** MEDIUM
- **Probability:** Medium
- **Impact:** Template colors don't apply, brand confusion
- **Description:** If templates.module.css has `button { background: blue !important }`, branding colors from 31.2 won't override it.
- **Mitigation:**
  - Template CSS uses CSS custom properties from 31.2: `background: var(--color-primary)`
  - NO `!important` flags in template CSS
  - Test each template type (standard/luxury/budget) with custom colors applied
- **Checklist:**
  - [ ] Template CSS uses CSS variables, not hardcoded colors
  - [ ] No `!important` in template CSS
  - [ ] Test: standard/luxury/budget each accept branding colors from 31.2

**R6: Missing Template Variant CSS Fallback**
- **Severity:** MEDIUM
- **Probability:** Low
- **Impact:** Broken layout if CSS class not applied
- **Description:** If `template_type` CSS class not properly applied (e.g., typo in root className), template defaults to unstyled layout.
- **Mitigation:**
  - Validate template_type is one of: 'standard', 'luxury', 'budget' (enum)
  - Fallback to 'standard' if invalid value
  - CSS includes base styles + variant overrides (not variant-only)
- **Checklist:**
  - [ ] Template_type enum validated (reject 'luxury-v2', etc.)
  - [ ] Base CSS styles apply to all templates
  - [ ] Variant CSS only overrides specific properties
  - [ ] Invalid template_type falls back to 'standard'

**R7: Performance - Property Filtering on Large Datasets**
- **Severity:** MEDIUM
- **Probability:** Low (only high-traffic orgs)
- **Impact:** Slow page loads for popular companies with 1000+ properties
- **Description:** If filtering properties in JavaScript (instead of database), booking page becomes slow.
- **Mitigation:**
  - Filter in database query: `WHERE organization_id = $1 AND id = ANY(featured_property_ids)`
  - Pagination support (limit 50 properties per page)
  - Caching: GET `/api/organizations/[orgSlug]/template` cached for 5 minutes
- **Checklist:**
  - [ ] Properties filtered in database, not in JavaScript
  - [ ] Pagination test: 500+ properties render with pagination
  - [ ] Caching headers set (Cache-Control: max-age=300)

### LOW

**R8: Backward Compatibility - Root Domain Templates**
- **Severity:** LOW
- **Probability:** Low
- **Impact:** Root domain behavior change, unexpected side effects
- **Description:** If root domain `lodgra.io/booking` uses templates feature, could affect all orgs globally.
- **Mitigation:**
  - Root domain booking page MUST NOT use organization_templates (returns null)
  - Root domain always shows all properties from all orgs (current behavior)
  - Code check: `if (orgSlug) { use template } else { show all }`
- **Checklist:**
  - [ ] Root domain (`lodgra.io/booking`) returns no template
  - [ ] Root domain booking page shows all org properties
  - [ ] No regression in root domain behavior
