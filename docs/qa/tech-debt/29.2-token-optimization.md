## Issue 1: Token Hash Lookup Performance

**Severity:** MEDIUM  
**Impact:** O(n) query for high-volume token verification  
**Current Implementation:**
```sql
SELECT * FROM cleaner_access_tokens 
ORDER BY created_at DESC 
LIMIT 100
-- Then linear search for matching hash in application code
```

**Optimization:**
```sql
-- Add unique index on token_hash
CREATE UNIQUE INDEX idx_cleaner_tokens_hash ON cleaner_access_tokens(token_hash);

-- Use direct lookup
SELECT id, cleaner_id, token_hash, expires_at, is_used, used_at 
FROM cleaner_access_tokens 
WHERE token_hash = $1 
LIMIT 1;
```

**Benefits:**
- O(1) lookup instead of O(n)
- Single database query instead of scan + iteration
- Reduced latency for high-frequency token verification
- Better scaling for enterprise deployments

**Effort:** 2-3 hours (add index, update query, test)  
**Timeline:** Sprint after 29.2 (non-blocking for MVP)

## Issue 2: Rate Limit Fallback Strategy

**Severity:** MEDIUM  
**Impact:** Could allow brute force if Upstash Redis unavailable  
**Current Implementation:**
```typescript
try {
  const { success, resetAfter } = await ratelimit.limit(`verify-token:${clientIp}`);
  if (!success) {
    return NextResponse.json({...}, { status: 429 });
  }
} catch (ratelimitError) {
  console.warn('Rate limit check failed:', ratelimitError);
  // Continues without rate limiting
}
```

**Optimization Options:**

**Option A: Circuit Breaker**
```typescript
const circuitBreaker = new CircuitBreaker({
  failureThreshold: 5,  // 5 failures
  timeout: 60000,       // 1 minute
  onOpen: () => {
    // Block verification temporarily
    return { allowed: false, reason: 'Rate limit service unavailable' };
  }
});
```

**Option B: Graceful Degradation**
```typescript
// If Redis down, use simple in-memory rate limiting
const inMemoryLimiter = new Map<string, number[]>();

function checkInMemoryLimit(ip: string): boolean {
  const now = Date.now();
  const attempts = inMemoryLimiter.get(ip) || [];
  const recentAttempts = attempts.filter(t => now - t < 15 * 60 * 1000);
  
  if (recentAttempts.length >= 5) return false;
  
  recentAttempts.push(now);
  inMemoryLimiter.set(ip, recentAttempts);
  return true;
}
```

**Benefits:**
- Prevents brute force even if external service fails
- Maintains security posture during incidents
- Clear incident alerting

**Effort:** 3-4 hours (implement, test failover)  
**Timeline:** Sprint after 29.2 (critical for production)  
**Priority Adjustment:** Recommend upgrading to HIGH for production deployment

## Issue 3: Suspicious Pattern Detection

**Severity:** LOW  
**Impact:** Can't detect fraud without pattern analysis  
**Current Implementation:**
- AC8 captures IP address and user agent on token use
- Data stored but not analyzed

**Optimization:**
```typescript
// In verify-token endpoint, after token use recorded
const suspiciousPatterns = await checkSuspiciousPatterns(tokenId, clientIp);

if (suspiciousPatterns.multipleIPs) {
  // Token used from 3+ different IPs in 15 minutes
  await captureToSentry({
    message: 'Suspicious token usage detected',
    level: 'warning',
    tags: { token_id: tokenId, distinct_ips: suspiciousPatterns.ips.length },
  });
}
```

**Detection Rules:**
- Multiple IPs per token (same token, 3+ different IPs)
- Rapid requests from different geographic regions
- Token usage from suspicious countries (if GeoIP available)

**Benefits:**
- Early fraud detection
- Compliance audit trail
- Automated alerts for security team

**Effort:** 4-5 hours (pattern detection, Sentry integration, testing)  
**Timeline:** Sprint 2-3 after 29.2 (optional for MVP)
